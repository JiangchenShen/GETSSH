## 前言：为什么需要 MOOVIER SUPREME {#前言}

### 问题陈述：现有视觉体系的天花板

GETSSH v2.0「黑曜石（Obsidian）」视觉语言完成了它的历史使命——它证明了一个 Electron 应用可以做到视觉上的精致与一致。但它本质上仍是一套**装饰性美学**，而非**结构性美学**。

它的局限体现在三个层面：

1. **玻璃感是伪造的**：`backdrop-filter: blur()` 只是对背后像素做了模糊处理，这是浏览器层面的软件滤镜。它没有光学衍射，没有材质折射，没有深度感。它让应用看起来有点透明，但不像「玻璃」，更像「毛玻璃贴纸」。

2. **布局是拼凑的**：各模块之间存在不规则的 `padding`、`margin`，侧边栏无法真正抵达系统窗口的物理边缘，各组件的内部间距缺乏统一的底层网格约束。这是一种"感觉上对齐"，而不是"数学上对齐"。

3. **动效是点缀的**：v2.0 的过渡动画是加在组件上的装饰，而不是从物理规律出发的设计。元素出现和消失，但没有重量感、没有惯性。

### MOOVIER SUPREME 的回答

MOOVIER SUPREME 不是一次界面改版，而是对 GETSSH 视觉体系的**底层重写**。它的核心命题是：

> **让每一个像素，都成为材质的一部分，而不是颜色的一部分。**

MOOVIER SUPREME 将建立在三块基石之上：
- **Liquid Glass（液态玻璃）**：用操作系统的原生材质 API，而不是 CSS 滤镜，来构建应用的视觉基底。
- **Absolute Grid（绝对网格）**：用严苛的数学对齐规则，取代直觉驱动的视觉排版。
- **Physical Motion（物理动效）**：用弹簧力学模型，取代线性的 `ease-in-out` 曲线。

本文档将按章节逐步定义上述三大基石的完整设计规范，并为工程实现提供明确的接口契约。

### 0.1 最高美学目标：Cinematic Immersion（电影级沉浸）

MOOVIER SUPREME 不是一次配色或布局上的迭代。它是一次**感知层面的根本革命**。

在绝大多数桌面工具软件的设计词汇里，「流畅」意味着「没有卡顿」，「美观」意味着「颜色好看」。这种对视觉品质的理解，停留在 Web 1.0 时代的工程视角——一种对「不难看」的消极追求。

MOOVIER SUPREME 拒绝这种定义。我们的参照系不是其他 SSH 客户端，甚至不是同类桌面工具。**我们的参照系是电影。**

具体而言，是顶级制作的电影在以下三个感知维度上建立的标准：

1. **时序的连续性（Temporal Continuity）**：伟大的电影从不存在「硬切」。每一个场景切换都是经过精心设计的过渡——推镜、溶解、淡出——目的是让观众的神经系统从未察觉到「断裂」的存在。UI 中每一个状态变更都是一次「场景切换」；MOOVIER SUPREME 将让这些切换消失在用户的感知之外。

2. **焦点的空间调度（Spatial Focus）**：伟大的电影通过景深控制观众的注意力——主体清晰锐利，背景散焦虚化。这不是技术特效，是叙事工具。MOOVIER SUPREME 将把这一工具移植到界面设计中：当用户专注于某一终端时，世界的其余部分应当优雅地退场，而不是喧宾夺主地共存。

3. **材质的物理真实（Physical Materiality）**：伟大的电影中的道具，即便是虚构的，也遵循物理规律——光线在玻璃上的折射、金属在侧光下的轮廓、布料在运动中的惯性。MOOVIER SUPREME 的界面材质，必须拥有同等的物理可信度：光从特定方向照来，照亮磁贴切面的折射线，投下具有景深感的阴影。

这是 MOOVIER SUPREME 视觉宪法的最高目标：**让 GETSSH 的每一个交互瞬间，都拥有电影帧的质感密度。**

### 0.2 第一定律：消除一切硬切（No Hard-Cuts）

**定义**：「硬切（Hard-Cut）」指 UI 状态从 A 到 B 的即时跳变——在同一帧内，旧状态消失，新状态出现，中间不存在任何过渡。在电影中，硬切是意图明确的叙事暴力；在 UI 中，硬切是设计师的失职。

MOOVIER SUPREME 宣布：**在整个 GETSSH 3.0 的 UI 范围内，任何可见状态的变更，都禁止以硬切方式呈现。**

#### 0.2.1 物理弹簧（Spring Physics）作为第一运动语言

所有涉及位置、尺寸、层级变化的 UI 动效，必须使用**物理弹簧（Spring）**而非传统的 CSS `transition`/`ease-in-out` 曲线。

两者的根本差异：

| 维度 | 传统曲线（`ease-in-out`）| 物理弹簧（Spring）|
|---|---|---|
| 数学模型 | 贝塞尔曲线（纯数学抽象）| 胡克定律（物理世界的弹性力学）|
| 持续时间 | 固定（设计师指定 `duration`）| 涌现（由质量、硬度、阻尼自然决定）|
| 外力响应 | 无（不能被打断并自然衔接）| 有（可在运动中途接受新的力输入，自然过渡至新目标）|
| 感知真实性 | 「像动画」（人工感）| 「像物体」（物理感）|

弹簧参数基准库遵循第 2 章 §2.3.1 的定义，此处仅声明其设计哲学层面的约束：

- **禁止使用 `duration` 参数来控制弹簧动效的长短**。弹簧的持续时间是其物理参数（`stiffness`、`damping`、`mass`）的自然结果，强行指定 `duration` 等同于在物理模拟中作弊。
- **所有弹簧动效必须是可中断的（Interruptible）**。若用户在动效进行中触发新操作，新的弹簧力必须以当前位置和速度为初始条件接管运动，而非等待旧动效结束后重新开始。

#### 0.2.2 定制贝塞尔曲线作为第二运动语言

对于**纯透明度变化（`opacity`）和颜色渐变**，弹簧物理模型不适用（透明度没有质量和弹性）。这类动效使用**定制三次贝塞尔曲线（Cubic Bézier）**作为过渡函数，但禁止使用 CSS 内置的 `ease`、`ease-in`、`ease-out`、`ease-in-out` 等预设值。

MOOVIER SUPREME 规定两条定制曲线：

| 曲线代号 | `cubic-bezier` 参数 | 设计意图 | 适用场景 |
|---|---|---|---|
| `CINEMATIC_IN` | `cubic-bezier(0.32, 0, 0.67, 0)` | 入场：慢启动，快到达，无缓出——像物体被突然赋予速度 | 面板/覆层入场，元素从静止开始运动 |
| `CINEMATIC_OUT` | `cubic-bezier(0.33, 1, 0.68, 1)` | 退场：快启动，慢收尾，带轻微余韵——像物体在阻力中减速至静止 | 面板/覆层退场，元素的透明度淡出 |
| `CINEMATIC_SNAP` | `cubic-bezier(0.87, 0, 0.13, 1)` | 极速响应，中段几乎线性，两端极短的缓冲——像高精度机械的咔哒感 | Toggle 切换、Tab 选中、状态灯变色等强反馈操作 |

> **`CINEMATIC_IN/OUT` 的电影学依据**：入场使用快速到达、无缓出的曲线，模仿摄影机推入镜头（Push-in）的加速感；退场使用快速启动、缓慢收尾的曲线，模仿画面溶解（Dissolve）的余韵感。两者组合，营造出有「呼吸感」的场景切换节奏。

### 0.3 第二定律：镜头级景深调度（Cinematic Focus Pulling）

**定义**：「对焦（Focus Pulling）」是电影摄影中的一种技术——通过动态调整镜头焦距，将视觉焦点从一个物体转移至另一个物体，同时让其他区域进入散焦模糊状态。

MOOVIER SUPREME 将此技术移植为**界面的注意力调度系统**。

#### 0.3.1 主焦点磁贴的定义

在任意时刻，GETSSH 界面中有且仅有**一个**主焦点磁贴（Primary Focus Tile）：

- 当用户在某一终端磁贴内进行键盘输入时，该磁贴自动成为主焦点磁贴。
- 当用户鼠标悬停于某一终端磁贴超过 `400ms`（防误触阈值）时，该磁贴成为主焦点磁贴。
- 当前的主焦点磁贴由 Zustand `nexus.focusTileId` 状态字段记录。

#### 0.3.2 景深模糊（Focus Blur）效果规范

当主焦点磁贴确立后，对**所有非焦点磁贴（Non-Focus Tiles）**施加以下视觉降压处理，营造影院熄灯般的空间沉浸感：

| 效果参数 | 非焦点磁贴目标值 | 主焦点磁贴目标值 | 过渡方式 |
|---|---|---|---|
| 亮度（`brightness`）| `0.60`（压暗至 60%）| `1.00`（全亮）| `CINEMATIC_OUT` 曲线，`300ms` |
| 饱和度（`saturate`）| `0.50`（去饱和至 50%）| `1.00`（全饱和）| `CINEMATIC_OUT` 曲线，`300ms` |
| 模糊半径（`blur`）| `0.6px`（极轻微散焦，不破坏文字可读性）| `0px`（锐利）| `CINEMATIC_OUT` 曲线，`200ms` |
| 不透明度（`opacity`）| `0.80`（轻度透明）| `1.00`（不透明）| `CINEMATIC_OUT` 曲线，`300ms` |
| 缩放（`scale`）| `0.992`（极轻微缩小，增强空间纵深感）| `1.000`（原始大小）| `SPRING_GENTLE` 弹簧 |

> **亮度降压的视觉心理学依据**：人类视觉系统对「亮度对比」极为敏感——当主体比周围环境亮 30~40% 时，注意力会被强制吸引至主体，无需任何运动线索。这就是电影院关灯后银幕显得更亮的原理。`0.60` 的亮度目标值，精确地创造了这一 40% 的亮度差，在不降低周围磁贴可读性的前提下，将注意力锚定在主焦点磁贴上。

#### 0.3.3 景深调度的豁免清单

以下元素**永远不受**景深降压处理影响，无论其是否处于焦点状态：

| 豁免元素 | 豁免原因 |
|---|---|
| 侧边栏磁贴（Sidebar Tile）| 导航工具，必须始终清晰可见，确保用户随时能够切换工作区 |
| 状态栏磁贴（Statusbar Tile）| 系统状态信息（CPU、内存、连接状态），降压可能导致关键警告被忽视 |
| RASP 安全告警覆层（Security Alert Overlay）| 安全告警必须以最高视觉权重呈现，不得被任何降压处理削弱 |
| 控制中心图层（CC Overlay）| 用户主动调出控制中心时，全域景深调度暂停，CC 内部有其独立的焦点逻辑 |
| 所有正在进行用户输入的覆层（Active Input Overlay）| 凡用户正在交互的覆层，均视同主焦点磁贴，豁免降压 |

#### 0.3.4 焦点切换的影院节奏

当主焦点从磁贴 A 切换至磁贴 B 时，过渡序列遵循以下严格的时序编排：

```
T+0ms    ：A 磁贴开始执行降压退出动效（brightness/saturate/opacity 降低，CINEMATIC_OUT 300ms）
T+0ms    ：B 磁贴开始执行焦点进入动效（scale 放大至 1.000，SPRING_GENTLE；brightness/opacity 升高，CINEMATIC_OUT 300ms）
T+150ms  ：B 磁贴的内发光亮度提升至焦点态峰值（静止态 × 1.5），持续 100ms 后回落至标准焦点态
T+300ms  ：过渡完成，两个磁贴的所有效果参数均已稳定至目标值
```

> **T+150ms 内发光脉冲的设计意图**：这是对「镜头对焦完成」瞬间的模拟——摄影机完成对焦的瞬间，主体轮廓会因焦点汇聚而显得更加锐利、边缘更亮。这个 100ms 的短促亮度脉冲，在视觉上传达的正是这一「咔哒，对上了」的清晰感。

### 0.4 第三定律：轮廓光刻画（Rim Lighting）

**定义**：「轮廓光（Rim Light / Back Light）」是电影布光中的经典三点布光法的第三盏灯——位于主体后方或侧后方，专门用于在主体轮廓上勾勒一道亮边，使主体从背景中「立体地剥离」出来，而不是「贴」在背景上。

MOOVIER SUPREME 的磁贴边缘高光，是对轮廓光的精确 UI 移植。**所有的磁贴边缘高光，在设计上不是「边框（Border）」，而是「光线在切面玻璃上的折射」。**

#### 0.4.1 轮廓光与边框的根本差异

| 维度 | 传统 CSS `border` | MOOVIER 轮廓光 |
|---|---|---|
| 设计语义 | 「这里有一条线」（几何界定）| 「这里有光照在玻璃切面上」（物理模拟）|
| 颜色来源 | 设计师指定的固定颜色 | 光源方向 + 材质折射率的函数 |
| 均匀性 | 四边等宽等色（默认）| 有方向性——光源方向的边缘更亮，背光侧更暗 |
| 实现方式 | `border: 1px solid color` | `box-shadow: inset`（严禁使用 `border` 属性）|
| 深度感 | 无（二维轮廓线）| 有（模拟 Z 轴厚度带来的切面折射）|

#### 0.4.2 全域光源方向约定（Global Light Source Convention）

MOOVIER SUPREME 规定全局唯一的**虚拟光源方向**，作为所有轮廓光计算的基准：

> **光源位置**：位于应用窗口的**左上角顶点**方向，与水平面成 `45°` 俯仰角，与垂直面成 `30°` 方位角。

此虚拟光源是整个 MOOVIER SUPREME 视觉系统的「物理锚点」——所有的高光位置、阴影方向、折射强度，都从这一虚拟光源出发，经过统一的向量计算得出。它确保了界面在任何状态下都具有**内在的光学一致性**——所有磁贴的高光和阴影看起来都像是处于同一个物理光环境下，而不是每个组件各自独立、光源四面八方。

#### 0.4.3 轮廓光在磁贴四边上的分布规范

基于上述全局光源方向（左上方 45°），轮廓光在磁贴四边上的分布遵循以下物理推导规律：

| 边缘位置 | 与光源的关系 | 轮廓光强度 | 实现方式 |
|---|---|---|---|
| **顶边** | 直接受光面（法线最接近光源入射角）| 最强：白色，透明度 `12%~18%`，宽度 `1px` | `box-shadow: inset 0 1px 0 rgba(255,255,255,0.15)` |
| **左边** | 侧受光面（法线与光源入射角约 30°）| 次强：白色，透明度 `6%~10%`，宽度 `1px` | `box-shadow: inset 1px 0 0 rgba(255,255,255,0.08)` |
| **右边** | 侧背光面（法线与光源方向相背约 30°）| 微弱：深色描边，透明度 `10%~15%`，宽度 `1px` | `box-shadow: inset -1px 0 0 rgba(0,0,0,0.12)` |
| **底边** | 完全背光面（法线完全背对光源）| 最暗：无内发光，由外部阴影的近场轮廓阴影承担视觉分离职责 | 底边不设 `inset box-shadow`；由 §2.2.3 的外层复合阴影系统处理 |

> **多重 `inset box-shadow` 叠加的技术约束**：以上四边的轮廓光效果，必须通过**单个 CSS `box-shadow` 属性内的多个值叠加**来实现（CSS 允许在同一个 `box-shadow` 声明中用逗号分隔多个值）。严禁为了实现轮廓光而在 DOM 结构中嵌套额外的 `<div>` 或伪元素，这会增加渲染层的复杂度并对性能造成不必要的影响。

#### 0.4.4 轮廓光的动态响应规范

轮廓光强度必须对以下用户交互动态响应，而不是始终维持静态值：

| 交互状态 | 顶边高光透明度调整 | 左边高光透明度调整 | 过渡方式 |
|---|---|---|---|
| 静止（Default）| 基准值（`12%~18%`）| 基准值（`6%~10%`）| — |
| 鼠标悬浮（Hover）| × `1.5`（提升至 `18%~27%`）| × `1.5`（提升至 `9%~15%`）| `CINEMATIC_SNAP`，`60ms` |
| 拖拽进行中（Dragging）| × `2.0`（提升至 `24%~36%`）| × `2.0`（提升至 `12%~20%`）| `SPRING_SNAPPY` |
| 主焦点态（Focus）| × `1.5`（同 Hover，但由景深系统驱动）| × `1.5` | `CINEMATIC_OUT`，`300ms` |
| 景深降压态（Non-Focus）| × `0.5`（衰减至基准值一半）| × `0.5` | `CINEMATIC_OUT`，`300ms` |

> **拖拽时的轮廓光爆亮（×2.0）的设计意图**：当用户将磁贴拾起（进入拖拽状态），模拟「将一块玻璃从桌面托起，使其正面更多地朝向光源」的物理行为——磁贴的受光角度改变，顶部和左侧切面接受到更多的光线，因此轮廓光强度显著提升。这为拖拽操作提供了强烈的「物体被拾起」的物理反馈。

### 0.5 电影级灵魂的统一时序宪章（Cinematic Timing Charter）

上述三大定律（无硬切、景深调度、轮廓光）共同运作时，其时序交互遵循以下统一宪章，确保在同一帧内不发生视觉指令的冲突与叠加混乱：

| 优先级 | 事件类型 | 时序处理规则 |
|---|---|---|
| 最高 | 安全告警触发 | 立即中断所有正在进行的景深调度和轮廓光动效，全域瞬间恢复至最高亮度状态，告警覆层以最高 Z-Material 层级弹出 |
| 高 | 撕裂事件（Tear-off，见第 3 章）| 目标磁贴的景深进入豁免状态，轮廓光进入拖拽态（×2.0），直至撕裂完成后退出豁免 |
| 中 | 焦点切换（磁贴 A → 磁贴 B）| 遵循 §0.3.4 中定义的 300ms 时序编排，不被其他低优先级事件打断 |
| 低 | 控制中心打开/关闭 | 打开时暂停景深调度（全域恢复基准亮度），关闭时恢复上一焦点状态的景深效果 |
| 最低 | 常规 Hover / 状态切换 | 遵循各自章节中定义的过渡参数，不触发全域效果调整 |

## 第 1 章：视觉基底与绝对网格 (Liquid Glass & Absolute Grid) {#第1章}


### 1.1 核心视觉底座：Liquid Glass（液态玻璃）

#### 1.1.1 设计哲学：从"滤镜"到"材质"的范式转移

传统的跨平台「玻璃」方案是在 Chromium 渲染层对背后的像素进行软件模糊，本质上是**图像后处理**。这种方式有两个根本性缺陷：

- **性能**：每次重绘都需要 GPU 对背景区域执行卷积模糊操作，在高分辨率屏幕或存在大量动态内容（如终端输出）的场景下，帧率会受到显著影响。
- **真实感**：软件模糊不会因光线角度变化而产生折射，不会有高光边缘，不会有深度层次。它只是"糊"，不是"透"。

MOOVIER SUPREME 的 Liquid Glass 方案，将彻底转向 **OS 原生材质 API 驱动**的路径：

| 平台    | 原生 API                         | 视觉效果描述                                                                                                |
| ------- | -------------------------------- | ----------------------------------------------------------------------------------------------------------- |
| macOS   | **NSVisualEffectView (Vibrancy)** | 系统级窗口背景混合（Window Blending），光线从底层桌面内容真实折射穿过，跟随系统暗/亮模式自动调整折射率。支持 `ultraThin`、`sidebar`、`headerView` 等多种材质规格。 |
| Windows | **Windows Mica / Mica Alt**     | 从用户的桌面壁纸采样并降噪混合至应用背景，产生与系统 Shell 统一的材质感。在 Windows 11 上最优，降级方案见第 5 章。|
| Linux   | **KDE Blur / GNOME Blur Effect** | 依赖 Wayland/X11 合成器的窗口背景混合 API，提供基础的桌面混合透明效果。降级为高质量的软件模糊方案。 |

> **工程接口约定**：Electron 主进程（`main/handlers/windowHandler.ts`）负责在窗口创建时设置操作系统级材质属性（`BrowserWindow.setVibrancy()` 等），并通过 IPC 通道响应用户的透明度调整请求。前端 CSS 层的职责不再是模拟玻璃效果，而是在 OS 原生材质之上叠加着色层（Tinting Layer）和边框光晕（Edge Highlight）。

#### 1.1.2 全局无级透明度系统（Transparency Slider）

用户对视觉偏好的控制权是设计中必须捍卫的核心价值之一。MOOVIER SUPREME 将提供一个**连续的、无级调节的全局透明度控制轴**，覆盖从「极致清透」到「完全着色」的全区间。

##### 透明度轴的两端定义

**极端 A——极致清透（Hyper-Transparent）**
- OS 原生材质完全掌权，应用背景内容几乎完全透穿（Show-Through）至桌面层。
- 着色层（Tinting Layer）透明度趋近于 `0%`。
- 适用场景：追求极致沉浸感的用户，在有大量图形/视频内容的桌面环境下效果最为惊艳。

**极端 B——完全着色（Fully Tinted）**
- OS 原生材质降为纯装饰（仅保留边框高光效果），着色层完全遮盖底层穿透内容。
- 着色层透明度趋近于 `100%`（深色模式下趋近于深炭黑色，浅色模式下趋近于浅雾白色）。
- 适用场景：在有合规要求或高保密需求的工作场景下，防止背后内容被旁观者看到；或在性能较低的设备上关闭混合效果。

##### CSS 变量接口契约

该系统的所有视觉参数，必须通过以下 CSS 自定义属性（Custom Properties）对外暴露，由主进程通过 IPC 动态注入 `:root` 的内联样式实现实时更新：

```
--glass-transparency: [0.0 ~ 1.0]
  含义：全局透明度主轴。0.0 = 完全着色，1.0 = 极致清透。
  驱动：由设置面板「透明度滑块」的值实时写入。

--glass-tint-opacity: [derived from --glass-transparency]
  含义：着色层的不透明度。由 --glass-transparency 反向计算得出。
  公式：--glass-tint-opacity = 1 - var(--glass-transparency) * 0.85

--glass-tint-color-dark: [HSL color value]
  含义：深色模式下的着色层基色。默认值：深炭黑（如 hsl(220, 15%, 8%)）。

--glass-tint-color-light: [HSL color value]
  含义：浅色模式下的着色层基色。默认值：高亮雾白（如 hsl(220, 20%, 97%)）。

--glass-edge-highlight-opacity: [0.0 ~ 1.0]
  含义：边框光晕（模拟玻璃边缘的高光反射）的不透明度。
  规则：在 --glass-transparency 较低时（着色模式）应等比降低，避免突兀的边框光晕出现在实色背景上。

--glass-blur-radius: [px]
  含义：软件 Fallback 模糊半径（用于 Linux 或不支持原生材质的场景）。默认值：24px。
```

> **重要约定**：任何 UI 组件绝对禁止在自身的 CSS 中硬编码 `backdrop-filter: blur(Xpx)` 或任何模糊值。所有与玻璃效果相关的视觉属性，必须且只能通过引用上述 CSS 变量来实现。这是维护全局透明度控制系统完整性的绝对红线。

#### 1.1.3 光学层级（Z-Material Stack）

与传统的 `z-index` 纯粹的层叠顺序不同，MOOVIER SUPREME 引入了**光学层级（Z-Material Stack）**的概念。每一个视觉层不仅有堆叠顺序，更有其材质属性和光线交互规则。

从底部到顶部的标准层级定义如下：

| 层级编号 | 层级名称           | 代号              | 描述                                                                                                       |
| -------- | ------------------ | ----------------- | ---------------------------------------------------------------------------------------------------------- |
| L0       | 系统桌面层         | `sys-desktop`     | 操作系统管辖范围，应用不可干预。是 Liquid Glass 折射的光源。                                                 |
| L1       | 原生材质层         | `native-material` | OS 原生 Vibrancy/Mica API 作用层。应用只设置属性，不渲染像素。                                               |
| L2       | 应用着色层         | `app-tint`        | 应用渲染的半透明着色覆盖层，通过 `--glass-tint-opacity` 控制浓度。是整个应用的"底色"。                        |
| L3       | 结构骨架层         | `app-chrome`      | 应用的不可见布局骨架（Grid 容器），定义所有区域的物理边界，但本身不渲染任何颜色或背景。                        |
| L4       | 区域材质层         | `region-material` | 各独立区域（侧边栏、终端区、状态栏等）的专属材质叠加，在 L2 基础上做局部的微调（如侧边栏轻微加深着色）。      |
| L5       | 内容层             | `content`         | 实际的 UI 内容（文字、图标、终端输出等）所在层。                                                            |
| L6       | 悬浮覆盖层         | `overlay`         | 模态弹窗、下拉菜单、Tooltip 等临时覆盖元素，拥有自己独立的 L1→L5 完整光学层级栈。                            |
| L7       | 系统通知层         | `sys-notification`| 安全告警覆层（RASP Red Alert 等），最高优先级，穿透一切常规 UI 层级。                                        |

> **核心规则**：每一层只与相邻层发生视觉交互。L4（区域材质层）只在 L2（应用着色层）之上叠加，绝不跨层与 L0 直接交互。这保证了全局 `--glass-transparency` 调整的可预测性——牵一发而动全身，而不是某个角落的组件独立于全局系统之外。

### 1.2 严苛的网格秩序（Absolute Grid Alignment）

#### 1.2.1 设计哲学：建筑蓝图，而非像素粘贴

传统终端应用的布局逻辑是**组件驱动**的：每个组件自己定义自己的边距，然后被「放进」一个容器里。这导致布局是拼凑出来的，存在大量的隐式间距和视觉噪音。

MOOVIER SUPREME 的布局逻辑是**网格驱动**的：应用存在一张唯一的、不可见的**绝对网格（Absolute Grid）**，这张网格由窗口的物理边缘出发，按照严格的数学规则划分全局空间。所有的 UI 区域，都是这张网格的**精确切割**，而不是放置在网格中的「盒子」。

这一哲学的根本转变体现在：

- **边缘就是边缘**：侧边栏的外缘，就是操作系统窗口的物理边缘。没有任何 `margin`、`padding`、`border-radius` 在它们之间存在。
- **间距来自网格，而非组件**：任何两个 UI 元素之间的视觉间距，必须是网格基准单位（`--grid-unit`）的整数倍，由网格列/行间距（`gap`）提供，而非由子元素自身的 `margin` 提供。
- **对称是公理，而非选项**：左侧侧边栏与右侧任何对称元素，必须在网格上拥有完全镜像的列定义。

#### 1.2.2 全局网格基准系统

MOOVIER SUPREME 采用**单一基准单位（Single Base Unit）**驱动整个布局系统，所有尺寸（间距、组件大小、字号行高等）必须是该基准单位的倍数。

**基准单位定义**

```
--grid-unit: 4px
```

这是整个视觉系统的最小原子单位。所有间距和尺寸均为其整数倍（如 4px, 8px, 12px, 16px, 24px, 32px, 48px, 64px…）。UI 中绝对禁止出现非 `--grid-unit` 整数倍的尺寸值（字号和行高等排版属性遵守单独的基准规则，见第 2 章）。

**全局应用级网格布局定义**

GETSSH 3.0 的应用根容器采用 CSS Grid 布局，定义如下语义列与行：

```
列定义（水平轴）：

  [window-edge-left]
      |
      |<--- 侧边栏（Sidebar） --->|
      |  宽度：--sidebar-width     |
      |  默认值：64px（Icon-Only） |
      |  展开值：220px（Labeled）  |
      |
  [sidebar-end / content-start]
      |
      |<-------- 主内容区（Main Content）-------->|
      |  宽度：1fr（自动填充剩余空间）             |
      |
  [content-end / window-edge-right]

行定义（垂直轴）：

  [window-edge-top]
      |
      |<--- 标题栏（Title Bar / Traffic Lights 区域）--->|
      |  高度：--titlebar-height                          |
      |  macOS 默认值：28px（取消默认 margin，Traffic Lights 原生位置）|
      |
  [titlebar-end / main-start]
      |
      |<--- 主工作区（Main Workspace）--->|
      |  高度：1fr                        |
      |
  [main-end / statusbar-start]
      |
      |<--- 状态栏（Status Bar）--->|
      |  高度：--statusbar-height   |
      |  默认值：24px               |
      |
  [statusbar-end / window-edge-bottom]
```

#### 1.2.3 边缘抵达规则（Edge-Flush Rules）

以下为 MOOVIER SUPREME 的**铁律**，工程实现中任何对以下规则的违反，都视为布局 Bug，必须修复：

**规则 E-1：侧边栏物理边缘抵达（Sidebar Edge Flush）**
- 侧边栏的 `left` 边界，必须与浏览器窗口的 `left: 0` 物理边缘零间距对齐。
- 侧边栏绝对禁止存在 `margin-left`、`padding-left`（内容的视觉内边距通过 `padding-block` 实现，但不影响背景区域）、`border-left` 或任何形式的外部容器包裹。
- 侧边栏背景（L4 区域材质层）必须以 `100%` 高度从 `window-edge-top` 延伸至 `window-edge-bottom`，完整覆盖所有行（包括标题栏和状态栏对应的行段）。

**规则 E-2：状态栏物理边缘抵达（Status Bar Edge Flush）**
- 状态栏的 `bottom` 边界，必须与窗口的 `bottom: 0` 物理边缘零间距对齐。
- 状态栏的 `left` 和 `right` 边界，必须分别与窗口的左右物理边缘对齐，不得有任何外部间距。

**规则 E-3：标题栏 Traffic Lights 区域统一管辖**
- macOS 的红绿灯（Traffic Light Buttons）所在的标题栏区域，必须与侧边栏在水平轴上完全对齐（即标题栏 Traffic Lights 区域宽度等于侧边栏宽度 `--sidebar-width`）。
- 侧边栏背景材质的光学效果（Vibrancy/Mica）必须连续穿过标题栏区域，视觉上形成一条完整的竖向「光学柱」，绝不允许标题栏与侧边栏之间出现任何视觉断层。

**规则 E-4：终端区域零边距（Terminal Zero-Margin）**
- 终端渲染区域（xterm.js 的 WebGL Canvas 容器）的外边缘，必须直接与其所在的 Pane 边界零间距对齐。
- 终端的「内边距（Padding）」感觉，必须通过 xterm.js 的 `padding` 配置项实现，而不是通过包裹容器的 CSS `padding` 实现。这保证了终端背景色（无论是透明还是实色）能够完整地与 Pane 区域无缝衔接。

#### 1.2.4 光线连续性原则（Light Continuity Principle）

Liquid Glass 效果的核心视觉价值在于：光线从桌面层（L0）穿过整个应用，产生统一的折射与透射。这一价值会被布局中的任何「视觉断层」所破坏。

**光线连续性要求的具体规则**：

1. **侧边栏玻璃连续性**：侧边栏的 L4 区域材质层，必须是 L2 着色层的纯粹叠加，而不是独立的新玻璃效果。视觉上，光线应该平滑地从主内容区穿过侧边栏分隔边界，侧边栏只是在同一片玻璃上施加了轻微的额外着色，而不是一块独立的玻璃被贴在旁边。

2. **分割线的非物质性**：侧边栏与主内容区之间的分隔，绝对禁止使用实色 `border`。如需提供视觉引导，应使用透明度极低（< 8%）的白色或黑色半透明线条，且该线条会随 `--glass-transparency` 的降低而等比淡化直至消失。

3. **圆角的物理限制**：窗口整体的圆角（Window Corner Radius），完全由操作系统决定，应用不得在 CSS 层强行为根容器设置 `border-radius`。应用内部的任何 UI 区域，使用 `border-radius: 0`（遵循整体的四边形零圆角美学，特定功能性组件如徽章除外）。

## 第 2 章：全域悬浮磁贴与物理光影雕刻 (Magnetic Tiles & Physical Shading)

### 2.1 设计哲学：一切皆磁贴（Everything is a Tile）

在 GETSSH v2.0「黑曜石」体系中，终端、侧边栏、面板等 UI 区域是作为**容器（Container）**存在的——它们是被填充内容的盒子，本身没有独立的物理属性。

MOOVIER SUPREME 颠覆这一模型，引入**磁贴（Tile）**作为系统最核心的视觉构件。磁贴不是容器，而是**漂浮在 Liquid Glass 基座之上的独立物理实体**。每一个磁贴都拥有：

- **自身的质量感**（通过阴影和边缘光晕体现的视觉重量）
- **与光源的关系**（通过顶/左边缘的内发光模拟玻璃切面的光学折射）
- **对外界输入的物理响应**（弹簧动效、按压形变）

在 GETSSH 3.0 中，以下所有 UI 区域，在规范层面均被视为独立的磁贴实体：

| 磁贴名称 | 描述 | 磁贴等级 |
|---|---|---|
| 终端磁贴（Terminal Tile） | 每一个 xterm.js 窗格，无论是全屏还是分屏 | 主磁贴（Primary） |
| 侧边栏磁贴（Sidebar Tile） | 应用的主导航侧边栏 | 主磁贴（Primary） |
| 监控面板磁贴（Monitor Tile） | 系统资源、RASP 告警等信息面板 | 次磁贴（Secondary） |
| 状态栏磁贴（Statusbar Tile） | 底部状态栏 | 辅助磁贴（Auxiliary） |
| 命令中心磁贴（Command Center Tile） | 全局命令中心（WelcomePane） | 悬浮磁贴（Floating） |
| 覆盖磁贴（Overlay Tile） | 下拉菜单、Tooltip、Modal 等临时元素 | 悬浮磁贴（Floating） |

> **磁贴等级的工程意义**：不同等级的磁贴，在光影参数（阴影扩散半径、内发光强度）和交互动效参数（弹簧硬度、位移幅度）上存在系统性差异。主磁贴拥有最完整的物理感，辅助磁贴的物理效果会相应收敛，避免整个界面「晃动过度」而产生视觉噪音。具体参数映射由第 4 章设计令牌统一定义。

---

### 2.2 磁贴的物理微观结构（Physical Micro-Structure）

每一个磁贴在视觉上由三个层次的光影元素构成，从内到外依次叠加。这三个层次共同决定了磁贴在 Liquid Glass 基座上的「剥离感（Lift Perception）」——即用户能直觉性地感知到磁贴是在玻璃表面上方漂浮的，而不是嵌入其中的。

#### 2.2.1 第一层：亮色内发光（1px Inner Glow）

**物理模拟对象**：光线以特定入射角照射到玻璃切面时，在切面的顶部和左侧边缘产生的高光反射。

每个磁贴的**顶部（Top）**和**左侧（Left）**边缘内侧，必须存在一条宽度为 `1px` 的半透明高光线。

| 参数 | 规范值 |
|---|---|
| 作用边缘 | 仅限顶边和左边 |
| 线条宽度 | `1px`，固定值，不随任何变量缩放 |
| 深色模式颜色 | `rgba(255, 255, 255, 0.10)` |
| 浅色模式颜色 | `rgba(255, 255, 255, 0.50)` |
| 实现方式 | `box-shadow: inset`，**禁止**使用 `border` 属性 |
| 对 `--glass-transparency` 的响应 | 不响应。内发光是磁贴自身物理属性，与背景透明度无关。 |

**主磁贴与次磁贴的参数差异**：

| 磁贴等级 | 深色模式内发光透明度 | 浅色模式内发光透明度 |
|---|---|---|
| 主磁贴 | 10% | 50% |
| 次磁贴 | 6% | 30% |
| 辅助磁贴 | 3% | 15% |
| 悬浮磁贴 | 15% | 60% |

#### 2.2.2 第二层：磁贴表面着色层（Tile Surface Tint）

**物理模拟对象**：磁贴自身材质对穿透光线的吸收与着色。这一层是磁贴在 Liquid Glass 基座上与背景进行区分的**主要视觉手段**。

| 参数 | 规范值 |
|---|---|
| 深色模式基础色 | `rgba(0, 0, 0, 0.20)` |
| 浅色模式基础色 | `rgba(255, 255, 255, 0.25)` |
| 实现方式 | `background-color`，必须使用 `rgba`，**禁止**使用不透明实色 |
| 对 `--glass-transparency` 的响应 | **弱响应**。着色模式时可适当加深，极致清透时保持基础值。 |

#### 2.2.3 第三层：复合边缘阴影（Composite Edge Shadow）

**物理模拟对象**：漂浮物体阻挡下方光源时的投影扩散。参考 WWDC 2026 Liquid Glass 材质规范。MOOVIER SUPREME 采用**复合三层阴影叠加**策略，确保在任意背景下的剥离感。

**三层阴影参数规范（以主磁贴为基准）**：

| 阴影层级 | 作用 | 扩散半径 | 偏移量 | 颜色 |
|---|---|---|---|---|
| 第 1 层 | 近场轮廓阴影 | `4px` | `Y: 2px` | `rgba(0, 0, 0, 0.30)` |
| 第 2 层 | 中场漫射阴影 | `12px` | `Y: 6px` | `rgba(0, 0, 0, 0.15)` |
| 第 3 层 | 远场环境阴影 | `32px` | `Y: 16px` | `rgba(0, 0, 0, 0.08)` |

> **关键原则**：三层阴影必须使用纯黑色 `rgba(0, 0, 0, alpha)`。带色相的阴影在不同桌面壁纸背景下表现极不稳定，唯有纯黑色具备跨背景普适性。

**次磁贴与辅助磁贴的参数收敛规则**：

| 磁贴等级 | 第 1 层透明度 | 第 2 层透明度 | 第 3 层透明度 |
|---|---|---|---|
| 主磁贴 | 0.30 | 0.15 | 0.08 |
| 次磁贴 | 0.20 | 0.10 | 0.05 |
| 辅助磁贴 | 0.12 | 0.06 | 0.00（省略） |
| 悬浮磁贴 | 0.35 | 0.18 | 0.10 |

### 2.3 物理心流交互（Physical Flow Micro-Interactions）

MOOVIER SUPREME 的所有磁贴交互动效，必须基于**弹簧物理模型（Spring Physics）**，而绝非基于线性时间曲线（`ease`, `ease-in-out`, `cubic-bezier` 等）。

#### 2.3.1 弹簧参数基准库（Spring Parameter Library）

所有磁贴交互的弹簧参数，必须从以下基准库中选取，**禁止**在组件层面自行定义任意数值的弹簧参数。

| 弹簧规格名称 | 适用场景 | 硬度（Stiffness） | 阻尼（Damping） | 质量（Mass） |
|---|---|---|---|---|
| `SPRING_SNAPPY` | 按钮点击、状态切换等即时反馈 | `400` | `30` | `1.0` |
| `SPRING_FLUID` | 磁贴悬浮上浮、面板滑入等主体动效 | `200` | `25` | `1.0` |
| `SPRING_GENTLE` | 全局透明度过渡、主题切换等慢速变化 | `80` | `20` | `1.0` |
| `SPRING_BOUNCY` | 成功/错误状态的强调反馈（需克制使用） | `300` | `15` | `1.2` |

#### 2.3.2 悬浮感应状态（Hover State）规范

**触发条件**：鼠标指针进入磁贴响应区域，且磁贴当前处于可交互状态。

| 变化维度 | 静止值（Rest） | 悬浮值（Hover） | 弹簧规格 |
|---|---|---|---|
| Y 轴位移 | `translateY(0px)` | `translateY(-2px)` | `SPRING_FLUID` |
| 内发光亮度 | 基准透明度 | 基准透明度 × 1.5 | `SPRING_FLUID` |
| 近场阴影（第 1 层）透明度 | 基准值 | 基准值 × 1.33 | `SPRING_FLUID` |
| 中场阴影（第 2 层）Y 偏移 | `Y: 6px` | `Y: 10px` | `SPRING_FLUID` |
| 远场阴影（第 3 层）Y 偏移 | `Y: 16px` | `Y: 24px` | `SPRING_FLUID` |

**作用域限制**：Y 轴位移变换**仅作用于磁贴自身的 Transform 层**，绝不影响相邻元素布局。必须通过 `transform` 属性施加，禁止修改 `margin`、`top`、`padding` 等文档流属性。

**侧边栏磁贴的悬浮例外**：侧边栏磁贴遵守规则 E-1（边缘抵达规则），**不执行 Y 轴位移**，悬浮效果仅限于内发光亮度提升和阴影加深，以避免与窗口边缘产生间隙。

#### 2.3.3 按压反馈状态（Active/Press State）规范

**按压物理模型**：模拟按压厚实玻璃平板的力学反馈——玻璃整体向按压方向轻微形变（缩小），释放后以弹簧动效快速恢复原位。

| 变化维度 | 悬浮值（Hover） | 按压值（Active） | 弹簧规格 |
|---|---|---|---|
| 缩放 | `scale(1.0)` | `scale(0.98)` | `SPRING_SNAPPY` |
| Y 轴位移 | `translateY(-2px)` | `translateY(0px)` | `SPRING_SNAPPY` |
| 内发光亮度 | 基准 × 1.5 | 基准 × 0.8 | `SPRING_SNAPPY` |
| 阴影扩散半径 | Hover 值 | 收缩至 Rest 值的 70% | `SPRING_SNAPPY` |

**释放行为**：`mouseup` 触发后，以 `SPRING_FLUID` 弹回 Hover 或 Rest 状态。弹回过程允许轻微超调（Overshoot），**禁止**人为截断弹簧动效的超调行为。

| 磁贴等级 | 按压缩放值 | 说明 |
|---|---|---|
| 主磁贴 | `scale(0.98)` | 标准按压幅度 |
| 次磁贴 | `scale(0.985)` | 略微收敛，避免大面积磁贴的剧烈形变 |
| 辅助磁贴 | `scale(0.99)` | 极微小的收敛 |
| 悬浮磁贴 | `scale(0.97)` | 略微放大按压感 |

### 2.4 极端环境下的文字可读性防御（Text Readability Defense）

MOOVIER SUPREME 将极端环境下的文字可读性视为**必须主动防御的可用性红线**，而非用户调整透明度后自行承担的视觉后果。

#### 2.4.1 可读性分区原则（Readability Zone Classification）

| 防御等级 | 区域类型 | 核心原则 |
|---|---|---|
| **P0——绝对清晰区** | 终端（xterm.js）、代码显示区、错误信息 | 任何情况下对比度不得低于 WCAG AAA 标准（7:1）。必须施加专属独立暗色滤镜遮罩。 |
| **P1——功能清晰区** | 命令行输入框、文件路径、连接参数表单 | 对比度不得低于 WCAG AA 标准（4.5:1）。需要局部加固着色。 |
| **P2——环境清晰区** | 侧边栏标签、状态栏信息、设置面板文字 | 对比度不得低于 WCAG AA 标准（4.5:1）。通过区域着色层自然保障。 |

#### 2.4.2 P0 级——终端专属暗色滤镜遮罩规范

终端区域必须拥有完全独立于全局 `--glass-transparency` 体系的**专属暗色滤镜遮罩层（Terminal Isolation Tint Layer）**，执行独立的非线性保底逻辑。

| 参数 | 规范值 |
|---|---|
| CSS 变量名 | `--terminal-isolation-opacity` |
| 默认值 | `0.50`（深色模式）/ `0.35`（浅色模式） |
| 最小值（安全底线） | `0.30`（深色模式）/ `0.20`（浅色模式）。无论用户如何调整透明度，遮罩绝不低于此下限。 |
| 用户可控性 | 可在「终端」→「背景遮罩」中独立调节，但下限受约束，**不允许调至 0**。 |
| 叠加方式 | `backdrop-filter: blur(var(--terminal-blur-radius))` + `background-color`，双重叠加 |
| `--terminal-blur-radius` 默认值 | `8px` |

**联动逻辑**：当 `--glass-transparency` 调高时，`--terminal-isolation-opacity` **逆向微幅提升**，以主动补偿背景复杂度提升带来的可读性损失。响应曲线为非线性——在低透明度区间（0.0~0.4）几乎不响应，在高透明度区间（0.7~1.0）显著提升。

#### 2.4.3 P1/P2 级加固与验收要求

**P1 加固**：高频文字交互区域背景额外叠加 `rgba(0, 0, 0, 0.15)` 的深色强化，该值为固定值，不随全局变量变化。

**P2 验收**：必须在以下极端场景进行专项 QA 测试：
- 纯白色壁纸 + `--glass-transparency: 0.9` + 浅色模式
- 纯红色（高饱和度）壁纸 + `--glass-transparency: 0.8` + 深色模式

若任何常规 UI 文字对比度低于 4.5:1，该区域必须升级为 P1 处理策略。

#### 2.4.4 对比度自动检测机制（Contrast Auto-Detection）

- 主进程通过 Electron `nativeTheme` API 或 OS 材质 API 感知桌面壁纸**主色调亮度（Luminance）**。
- 检测到高风险环境时，主进程通过 IPC 触发渲染进程将 `--terminal-isolation-opacity` 自动提升至安全值（上限 `0.70`）。
- 该提升为**临时性**操作，不覆写用户偏好，切换至安全环境后自动恢复。
- 过渡效果使用 `SPRING_GENTLE` 弹簧规格，确保平滑过渡。

## 第 3 章：多窗口撕裂与跨次元状态交接 (Multi-Window Tearing & State Handoff)

### 3.1 设计哲学：突破玻璃边框（Breaking the Glass Frame）

传统多窗口模型中，"撕出（Tear-off）"一个面板，意味着销毁当前实例并在新的操作系统原生窗口中重建一个全新的渲染进程——这是一种彻底的断裂操作，用户在视觉上感受到的是「消失→重新出现」的硬切换，而非物理的「撕脱」。

MOOVIER SUPREME 的多窗口撕裂设计，其核心命题是：

> **让用户感受到的，是真实的物理撕裂——磁贴真的从主程序的玻璃框架上被剥下来，飘到了系统桌面。**

实现这一命题需要在三个技术层面同时发力：
- **前端视觉层**：用 Framer Motion 物理引擎完整模拟磁贴在撕裂前、中、后的物理运动状态。
- **进程管理层**：用空壳窗口池化（Hollow Window Pool）技术，在用户「撕」的那一刻，让独立原生窗口以几乎零延迟的速度出现在正确的屏幕坐标上。
- **数据层**：用零拷贝共享内存（Zero-Copy SharedArrayBuffer）技术，确保无论磁贴在哪个窗口中渲染，它读取的都是 Rust PTY 引擎维护的同一份状态流，不存在任何数据复制和状态撕裂。

### 3.2 物理级拖拽与视界越界（Framer Motion Drag & Boundary）

#### 3.2.1 全局拖拽能力分类

GETSSH 3.0 中，并非所有磁贴都支持脱离主窗口的撕裂操作。按照拖拽能力，磁贴被分为三类：

| 能力等级 | 支持的磁贴类型 | 描述 |
|---|---|---|
| **全局拖拽（Global Drag）** | 终端磁贴、监控面板磁贴、命令中心磁贴 | 可在主窗口内自由拖拽重排，且支持越界撕裂至独立原生窗口 |
| **局部拖拽（Local Drag）** | 覆盖磁贴（Modal/Dropdown） | 仅在主窗口内拖拽，触达主窗口边界后强制回弹，不触发越界逻辑 |
| **固定（Fixed）** | 侧边栏磁贴、状态栏磁贴 | 不支持任何拖拽操作，其位置由绝对网格（第 1 章规范）永久锁定 |

#### 3.2.2 窗口内拖拽的物理行为规范

当用户对「全局拖拽」等级的磁贴发起拖拽操作时（`mousedown` → `mousemove`），拖拽行为必须完全由 Framer Motion 物理引擎接管，不得使用任何静态 CSS Transition 或 JavaScript 线性插值来模拟位置更新。

**窗口内拖拽物理参数规范**：

| 物理参数 | 规范值 | 说明 |
|---|---|---|
| 拖拽惯性（Drag Inertia） | 启用 | 用户松手后，磁贴以当前速度向量继续滑行，并以 `SPRING_FLUID` 弹簧规格减速至最终落点 |
| 滑行阻力系数（Damping） | `0.85`（归一化，1.0 = 立刻停止） | 在视觉上产生类似「玻璃在光滑桌面上滑行」的阻尼感 |
| 角速度旋转（Tilt on Drag） | `±4deg`（基于拖拽速度向量动态计算） | 拖拽过程中，磁贴根据当前运动速度向量轻微倾斜，速度越快倾斜越大，模拟物体快速移动时的空气阻力形变 |
| 阴影实时更新（Shadow on Drag） | 启用 | 拖拽过程中，复合边缘阴影的 Y 轴偏移量实时跟随拖拽速度增减（速度越快，阴影越深越远），增强高度感 |
| 内发光实时亮化（Glow on Drag） | 启用 | 拖拽状态下，内发光亮度提升至静止态的 `2.0` 倍，模拟「被拾起」的光学效果 |
| 弹簧规格（落定时） | `SPRING_FLUID` | 用户松手或拖拽被取消时，磁贴以此弹簧规格弹回最近的合法落点 |

**拖拽期间的 Z 轴优先级**：被拖拽的磁贴必须强制提升至 Z-Material Stack 的 L6 层（悬浮覆盖层），确保在穿越其他磁贴上方时始终保持视觉置顶。拖拽结束后，Z 轴恢复至该磁贴的原始等级。

#### 3.2.3 越界侦测状态机（Boundary Detection State Machine）

「越界侦测」是触发多窗口撕裂流程的门控机制。前端必须在拖拽过程中对以下条件进行持续、实时的判断：

**触发越界侦测的复合条件（以下条件必须同时满足）**：

| 条件编号 | 条件描述 | 检测方法 |
|---|---|---|
| C-1 | 磁贴的碰撞边界框（Bounding Box）与主窗口的任意一条物理边缘产生重叠 | 实时计算磁贴位置与 `window.innerWidth` / `window.innerHeight` 的差值 |
| C-2 | 鼠标指针的当前坐标（相对于屏幕绝对坐标）已越过主窗口边缘，进入主窗口外的区域 | 对比 `MouseEvent.screenX/Y` 与主窗口的 `outerWidth/outerHeight` 和屏幕坐标 |
| C-3 | 当前拖拽速度向量的方向分量指向越界方向（即用户正在「向外拽」） | 对 Framer Motion 的 `drag velocity` 进行方向分量计算 |
| C-4 | 越界趋势持续时间超过 `80ms` 阈值（防误触抖动） | 时间戳差值计算 |

**越界侦测状态定义**：

```
状态 A：IDLE（空闲）
  → 无拖拽操作进行中
  → 条件全不满足

状态 B：DRAGGING（窗口内拖拽）
  → 拖拽进行中
  → C-1 或 C-2 不满足

状态 C：BOUNDARY_APPROACHING（趋近边界）
  → C-1 满足，C-2/C-3/C-4 不全满足
  → 视觉反馈：磁贴边缘阴影颜色切换为品牌色调，内发光频率轻微「脉冲」，提示用户继续拖拽可触发撕裂

状态 D：TEAR_ARMED（撕裂就绪）
  → C-1、C-2、C-3、C-4 全部满足
  → 撕裂交接状态机进入预备阶段
  → 视觉反馈：磁贴的边缘高光爆闪一次（50ms 内亮度从基准骤升至 3.0× 再恢复），告知用户「即将撕出」

状态 E：TEARING（撕裂进行中）
  → 用户松手，状态 D 触发撕裂事件
  → 执行 §3.3 中定义的「瞬间夺舍」全流程
```

> **状态 C 的设计意义**：在 BOUNDARY_APPROACHING 状态下提供视觉预警，是防止用户「意外撕出」的关键防误设计。用户可以在看到脉冲信号后减速或改变方向，系统将优雅地回退至 DRAGGING 状态，不会触发任何撕裂动作。

### 3.3 瞬间夺舍：空壳窗口池化技术（Hollow Window Pool）

#### 3.3.1 设计背景与核心约束

**绝对禁止的操作**：当撕裂事件触发时，主进程绝不允许执行以下任何操作：

| 禁止操作 | 原因 |
|---|---|
| 调用 `new BrowserWindow()` 创建新的完整渲染进程 | 在 macOS 上，完整 Chromium 渲染进程的冷启动时间约为 400ms~1200ms，远超视觉容忍阈值（< 50ms），会造成明显的视觉断层感 |
| 同步阻塞主进程等待新窗口就绪 | 会导致 IPC 通信队列堵塞，主窗口 UI 帧率骤降 |
| 从主窗口传输任何序列化的终端历史数据至新窗口 | 见 §3.4 零拷贝机制，此为架构层面的强制约束 |

#### 3.3.2 空壳窗口的定义与预热规范

**空壳窗口（Hollow Window）**的定义：一个**完全透明**、已完成 Chromium 渲染引擎初始化、加载了最小化启动脚本（仅包含 SharedArrayBuffer 连接逻辑和基础 CSS 变量注入）但**尚未渲染任何 UI 内容**的原生 `BrowserWindow` 实例。

**预热时机**：

| 事件 | 主进程行为 |
|---|---|
| 应用启动完成，主窗口首次渲染结束后 `3000ms` | 在后台异步创建第 1 个空壳窗口（优先级：`idle`，使用 `process.idle` 调度，不抢占主线程资源） |
| 第 1 个空壳窗口就绪后 `5000ms` | 在后台异步创建第 2 个空壳窗口（维持池容量为 2） |
| 空壳窗口被消耗（即执行了一次撕裂夺舍）| 延迟 `2000ms` 后补充一个新的空壳窗口，将池容量恢复至 2 |

**空壳窗口的资源规范**：

| 资源参数 | 规范值 | 说明 |
|---|---|---|
| 窗口透明度 | `opacity: 0`，窗口透明色，且 `show: false` | 完全不可见，不出现在任务栏或 Mission Control |
| 内存占用目标 | < 30MB / 个 | 仅包含 Chromium 渲染上下文，不加载任何业务逻辑或字体资源 |
| CPU 占用目标 | < 0.5% | 空壳窗口不执行任何定时器、动效或网络请求 |
| 预加载脚本范围 | 仅注入 SharedArrayBuffer 连接探针 + CSS 变量初始化代码 | 确保夺舍后能够立即接管全局设计令牌（透明度、主题色等）|
| 调度优先级 | `idle`（最低优先级，主进程空闲时才执行） | 严格避免与主窗口的关键渲染路径竞争资源 |

#### 3.3.3 视觉欺骗与夺舍全流程规范

当前端侦测到状态机进入 `TEARING` 状态（用户松手，撕裂条件全部满足）后，以下步骤必须严格按顺序、在规定时间内完成：

**Phase 0：预备（在状态 D `TEAR_ARMED` 期间即开始，用户尚未松手）**

| 步骤 | 执行方 | 操作 | 时间约束 |
|---|---|---|---|
| P0-1 | 前端 | 向主进程发送 `TEAR_ARM` IPC 预警信号，携带目标磁贴的 `tileId` | 立即（状态进入 D 时） |
| P0-2 | 主进程 | 从空壳窗口池中预选一个可用空壳窗口，完成 SAB（SharedArrayBuffer）连接握手，将其置于「就绪待命」状态 | < 20ms |
| P0-3 | 主进程 | 将就绪空壳窗口移动至当前鼠标的屏幕绝对坐标附近（但保持 `opacity: 0`） | < 5ms |

**Phase 1：撕裂触发（用户松手的瞬间，T=0ms）**

| 步骤 | 执行方 | 操作 | 时间约束 |
|---|---|---|---|
| T1-1 | 前端 | 将主窗口中目标磁贴的 React 组件设为 `opacity: 0`，同时停止其 Framer Motion 物理计算 | T+0ms（同帧执行） |
| T1-2 | 前端 | 向主进程发送 `TEAR_EXECUTE` IPC 信号，携带磁贴当前的屏幕绝对坐标（`screenX, screenY`）、尺寸（`width, height`）和当前拖拽速度向量 | T+0ms |
| T1-3 | 主进程 | 将预选空壳窗口的尺寸精确设置为磁贴尺寸，将其坐标精确设置为磁贴的屏幕绝对坐标 | T+0ms ~ T+8ms |
| T1-4 | 主进程 | 向空壳窗口注入目标磁贴的渲染指令（`tileId`、设计令牌、Workspace 上下文） | T+5ms ~ T+15ms |
| T1-5 | 主进程 | 将空壳窗口的 `opacity` 从 `0` 设置为 `1`，`show: true` | T+16ms ~ T+24ms |

> **视觉欺骗成立的时间窗口**：人眼对「同位置画面消失再出现」的感知阈值约为 **50ms**。整个 Phase 1 流程的总耗时目标为 **< 24ms**，确保在人眼感知阈值的一半以内完成视觉交接，用户感知到的是「磁贴突破了玻璃框飞出去了」，而不是「消失了又出现了」。

**Phase 2：飞行惯性继承（T=24ms 起）**

空壳窗口在屏幕坐标上完成出现后，必须继承磁贴在 `TEARING` 触发瞬间的速度向量，执行惯性滑行：

| 参数 | 规范值 |
|---|---|
| 初始速度 | 继承自 `TEAR_EXECUTE` 信号携带的拖拽速度向量（`vx`, `vy`） |
| 惯性衰减 | 以 `SPRING_FLUID` 弹簧规格在屏幕坐标系中减速至静止 |
| 屏幕边界反应 | 触达屏幕物理边缘时，速度法线分量归零（沿边缘滑行），切线分量保留 `70%`（模拟玻璃碰到桌面边缘的滑擦感） |
| 最终停靠 | 惯性衰减后静止在当前坐标，**不自动吸附**至屏幕网格或其他任何「对齐区域」 |

**Phase 3：主窗口布局收口（T=0ms 同步进行）**

主窗口在目标磁贴消失后，必须对其所占据的空间进行收口处理：

| 场景 | 收口行为 |
|---|---|
| 目标磁贴是多分屏布局中的某一个叶节点 | Nexus Core 状态机接到 `nexus:close-tab` 信令，将该叶节点从二叉树中移除，相邻叶节点以 `SPRING_GENTLE` 弹簧规格扩张填充空缺 |
| 目标磁贴是当前唯一的全屏叶节点 | 该叶节点位置替换为 WelcomePane（Command Center），以 `SPRING_FLUID` 弹入 |
| 撕裂被取消（用户在 `BOUNDARY_APPROACHING` 状态回撤） | 磁贴原位恢复 `opacity: 1`，Nexus Core 状态保持不变，预选的空壳窗口恢复待命状态 |

### 3.4 零拷贝渲染机制（Zero-Copy Viewport Chunking）

#### 3.4.1 架构核心约束：禁止在 JS 堆内保存终端历史

GETSSH 的终端历史（PTY 输出流）是应用中**数据量最大、增长最快**的数据结构。在高频输出场景（如 `tail -f` 大型日志、大量编译输出）下，终端历史缓冲区可能在短时间内达到数十 MB 甚至数百 MB。

**绝对禁止的内存模式**：

| 禁止行为 | 后果 |
|---|---|
| 在子窗口的 JS 堆中完整复制主窗口的终端历史缓冲区 | 子窗口内存峰值与主窗口等量叠加，100 个子窗口 = 100× 内存膨胀 |
| 通过 Electron IPC 将终端历史序列化后传输至子窗口 | IPC 通道的序列化（JSON / Structured Clone）将引发严重的 CPU 峰值和主进程阻塞 |
| 子窗口内独立维护一份 xterm.js 缓冲区副本并持续同步 | 状态不一致（撕裂感），且存在双倍的内存和 CPU 开销 |

#### 3.4.2 单一 PTY 状态流架构（Single PTY State Stream）

GETSSH 3.0 规定，PTY 的输出状态由且仅由 **Rust PTY 引擎**（`rust-core/nexus-core` 中的 PTY 状态机模块）作为唯一权威来源（Single Source of Truth）进行维护。

数据流向定义如下：

```
[远端 SSH 服务器 / 本地 PTY 进程]
         ↓ 原始字节流（binary stream）
[Rust PTY 引擎（运行于主进程 Rust 线程池）]
  职责：
  ① 维护完整的 PTY 输出历史环形缓冲区（Ring Buffer），上限由配置决定
  ② 维护当前渲染视口（Viewport）的状态快照：光标位置、当前屏幕行内容、颜色属性等
  ③ 将视口快照实时写入 SharedArrayBuffer（SAB）
         ↓ SharedArrayBuffer（零拷贝共享内存，无序列化开销）
[主窗口 xterm.js Canvas] ← SAB Reader（直接读取，不持有独立副本）
[子窗口 1 xterm.js Canvas] ← SAB Reader（直接读取，不持有独立副本）
[子窗口 N xterm.js Canvas] ← SAB Reader（直接读取，不持有独立副本）
```

#### 3.4.3 SharedArrayBuffer 数据结构规范

SAB 中存储的是**渲染视口状态快照（Viewport State Snapshot）**，而不是完整的 PTY 历史。其数据结构须满足以下规范：

| 数据字段 | 类型 | 说明 |
|---|---|---|
| `viewport_rows` | `Uint16` | 当前渲染视口行数 |
| `viewport_cols` | `Uint16` | 当前渲染视口列数 |
| `cursor_row` | `Uint16` | 光标当前行（相对于视口顶部） |
| `cursor_col` | `Uint16` | 光标当前列 |
| `cell_data[]` | `Uint32Array` | 视口内每个字符格（Cell）的编码，含字符码位（Unicode Code Point）、前景色索引、背景色索引、属性标志位（Bold/Italic/Underline/Blink）打包至单个 32 位整数 |
| `dirty_flag` | `Int32`（Atomics 操作） | 由 Rust 写端使用 `Atomics.store` 原子写入版本号；JS 读端使用 `Atomics.load` 检测变更，避免无效渲染 |
| `scroll_offset` | `Int32` | 当前视口相对于历史缓冲区顶部的滚动偏移量（行数） |

> **`dirty_flag` 的工程意义**：JS 端的 xterm.js Canvas 刷新循环，必须首先对比 `dirty_flag` 的当前值与上一帧读取到的值。若相同，则跳过本帧全部重绘操作（Zero Render Cost）；若不同，则从 SAB 中读取最新的 `cell_data[]` 并提交渲染。这一机制将 PTY 空闲时（无输出）的 GPU 渲染开销降至绝对零。

#### 3.4.4 历史滚动的零拷贝请求机制

当用户在子窗口中进行**滚动回溯**（即查看早于当前视口的历史输出）时，子窗口不得在本地缓存历史数据。历史数据的获取必须通过以下「视口请求」机制实现：

1. 用户在子窗口滚动，触发 `scroll_offset` 更新请求（`IPC: pty:viewport-request`，携带 `sessionId` 和新的 `scroll_offset` 值）。
2. 主进程 Rust PTY 引擎接收到视口请求后，根据 `scroll_offset` 从历史环形缓冲区中提取对应范围的行数据，写入对应的 SAB `cell_data[]` 区域，并原子递增 `dirty_flag`。
3. 子窗口的 JS 读端检测到 `dirty_flag` 变更，从 SAB 中读取更新后的 `cell_data[]` 并执行渲染。
4. 子窗口的 JS 层**不持有任何历史行数据的本地副本**。每次滚动都是一次「视口位置改变」的轻量请求，而非「数据拷贝」操作。

> **上限与降级**：Rust 历史环形缓冲区的行数上限由用户的「回滚行数」配置项决定（默认 10,000 行，同 v2.0 规范）。超出上限的历史将被环形覆盖，不可找回。该上限为整个 Session 的全局唯一缓冲区，所有子窗口共享，不随子窗口数量倍增。

#### 3.4.5 子窗口回归主窗口（Re-Docking）规范

当用户将游离的子窗口拖拽至主窗口边缘（磁吸区域），并选择将其回归主窗口时，执行逆向的夺舍流程：

| 步骤 | 执行方 | 操作 |
|---|---|---|
| D-1 | 子窗口前端 | 侦测到拖拽坐标与主窗口边缘的磁吸区域重叠，进入 `REDOCK_ARMED` 状态，触发边缘磁吸动效（子窗口被轻微「吸向」主窗口边缘方向） |
| D-2 | 子窗口前端 | 用户松手，向主进程发送 `REDOCK_EXECUTE` 信令，携带目标 `tileId` 和目标插入位置（在主窗口 Nexus 分屏树中的落点） |
| D-3 | 主进程 | 将 Nexus Core 状态树中的目标落点插入新叶节点，通知主窗口前端以 `SPRING_FLUID` 弹出新磁贴 |
| D-4 | 主窗口前端 | 新磁贴从对应屏幕坐标以 `SPRING_FLUID` 规格飞入主窗口目标位置，`opacity` 从 `0` 渐入至 `1` |
| D-5 | 主进程 | 子窗口 `show: false`，`opacity: 0`，回归空壳窗口池待命 |
| D-6 | 全局 | SAB 连接关系不变更，该 Session 的 PTY 数据流仍由 Rust 引擎统一维护 |

> **SAB 连接不变更的工程意义**：无论磁贴是在主窗口、子窗口、还是在飞行中，它读取的始终是 Rust 引擎写入的同一个 SAB 地址段。撕出与回归操作，本质上只是「哪个 Canvas 去读这个 SAB」的切换，而不是任何数据的搬运。

## 第 4 章：全景磁贴仪表盘与控制中心重制 (Modular Dashboard & Control Center Overhaul)

### 4.1 设计哲学：控制中心不是「页面」（Control Center is Not a Page）

GETSSH v2.0 的偏好设置体系沿用了桌面软件自 1990 年代以来从未根本改变过的模式：**左侧导航树 + 右侧长表单**。这一模式的设计出发点是「完备性」——确保每一个配置项都能被找到。但它的代价是极其高昂的认知负担：用户必须在脑中维持一张导航树的心理模型，才能知道「我想改的那个东西在哪个二级子菜单里」。

MOOVIER SUPREME 宣告这一模式的终结。控制中心将被彻底重新定义：

**旧定义**：一个从主界面跳转进入的独立「设置页面」，包含线性排列的导航树和表单。

**新定义**：一块直接覆盖在主 Liquid Glass 基座之上的**大尺寸悬浮图层（Control Center Overlay）**，其上排列着多个尺寸不一、功能自洽的**设置磁贴（Setting Tile）**。用户不需要「进入」控制中心，控制中心从主界面「升起来」，展示完毕后「落回去」，始终是主界面的一部分，而非一个目的地。

这一转变的核心设计逻辑，来自三个观察：

1. **频次分布极度不均**：用户 80% 的配置操作集中在 20% 的配置项上（字体大小、透明度、主题色、AI 密钥）。传统设置页面对所有配置项一视同仁，是对高频操作的惩罚。
2. **物理化操作比表单更直觉**：拖拽一个透明度滑块并直接看到背后 UI 跟随变化，比在一个输入框里输入数字然后点击「应用」要快 5 倍、准确 10 倍。
3. **信息的空间分布优于线性列表**：当配置项以「卡片网格」而非「导航树」的形式展现时，用户可以通过视觉扫描（而非记忆导航路径）找到目标。

### 4.3 设置磁贴分类体系（Setting Tile Classification System）

#### 4.3.1 磁贴尺寸规格（Tile Size Spec）

借鉴 iPadOS 控制中心的「大小磁贴混排」哲学，GETSSH 3.0 控制中心的设置磁贴按信息密度和操作频次分为三种标准尺寸：

| 尺寸规格 | 网格占用 | 像素约值 | 适用内容 |
|---|---|---|---|
| **S（小）** | 1列 × 1行 | `~160×160px` | 单一开关类操作（如「启用毛玻璃」Toggle、「跟随系统主题」Toggle） |
| **M（中）** | 1列 × 2行 | `~160×336px` | 单维度滑块（如透明度滑块、字号滑块），卡片表面直接承载滑块交互 |
| **L（大）** | 2列 × 2行 | `~336×336px` | 多维度信息展示（如主题色板、RASP 安全等级雷达图、AI 密钥绑定状态面板） |
| **XL（特大）** | 3列 × 2行（全宽） | `~528×336px` | 仅用于最高频操作的「英雄磁贴（Hero Tile）」，如字体实时预览卡 |

**混排规则**：网格布局引擎应优先填充 L 和 XL 磁贴，确保视觉上的重量感均衡。所有磁贴之间的间距统一使用 `--grid-unit * 4`，不得使用不规则间距。

#### 4.3.2 标准设置磁贴目录（Standard Setting Tile Catalog）

以下为控制中心的标准磁贴集合，每个磁贴的尺寸、功能域和主交互模式均已预定义：

| 磁贴名称 | 尺寸 | 功能域 | 表面主交互 | 下钻内容 |
|---|---|---|---|---|
| **透明度控制（Transparency）** | M | 视觉基底 | 竖向滑块，实时更新 `--glass-transparency` | 无（M 尺寸即为完整交互） |
| **主题色板（Theme Color）** | L | 视觉基底 | 色环 + 预设色块点击，实时更新 `--primary-color` | 自定义 HEX/HSL 值输入 |
| **深色/浅色/自动（Appearance）** | S | 视觉基底 | 三段切换按钮 | 无 |
| **毛玻璃开关（Glass Effect）** | S | 视觉基底 | 单一 Toggle | 无 |
| **字体与排版（Typography）** | XL | 终端体验 | 字体族选择器 + 字号滑块，右侧实时预览渲染效果 | 等宽字体高级设置（行高、字距、连字开关） |
| **终端背景遮罩（Terminal Tint）** | M | 终端体验 | 竖向滑块，实时更新 `--terminal-isolation-opacity` | 无 |
| **AI 密钥绑定（AI Keys / BYOK）** | L | AI 中心 | 显示当前绑定状态（已绑定 / 未绑定），各模型供应商的连接状态灯 | 密钥输入、端点配置、连接测试 |
| **RASP 安全等级（Security Level）** | L | 安全中心 | 四段选择器（SAFE / STRICT / NORMAL / DEV），当前等级高亮，切换时弹出 Touch ID 确认 | 六大安全屏障详情、Watchdog 实时状态 |
| **插件管理（Plugins）** | L | 插件生态 | 已安装插件列表缩略图，显示激活/停用状态 | 插件详情、权限审查、卸载 |
| **已知主机（Known Hosts）** | M | 安全中心 | 数字徽章显示当前受信主机数量 | 受信主机逐条列表，支持撤销 |
| **连接代理（Network Proxy）** | S | 网络 | 代理开关 Toggle，开启时显示代理类型标签 | 代理类型、地址、端口配置 |
| **Watchdog 状态（Watchdog）** | S | 安全中心 | 实时心跳状态指示灯（绿色闪烁 = 正常）| Watchdog 延迟可视化 |
| **锁屏超时（Auto-Lock）** | S | 隐私 | 当前超时时长数字显示 | 超时时长设置、锁屏立即触发按钮 |
| **语言（Language）** | S | 国际化 | 当前语言国旗/缩写 | zh-CN / en-US 切换 |
| **关于与更新（About）** | M | 系统 | 版本号、构建哈希展示，更新检查状态 | 完整运行环境信息（Electron / Node / Chrome / 架构）|

> **磁贴顺序原则**：控制中心网格的渲染顺序，遵循「高频优先、危险靠后」的排布规则。透明度、主题色、字体等日常高频磁贴位于网格左上角；AI 密钥、RASP 等级等低频高权限磁贴位于右下角区域。

### 4.4 设置磁贴的两种交互模式（Two Interaction Modes）

#### 4.4.1 表面交互模式（Surface Interaction）

**定义**：用户可以直接在磁贴的表面完成操作，无需进入任何子层级或弹出任何附加 UI。这是控制中心「零步骤操作」哲学的核心实现。

**适用磁贴**：透明度控制（M 尺寸滑块）、终端背景遮罩（M 尺寸滑块）、深色/浅色/自动（S 尺寸三段切换）、毛玻璃开关（S 尺寸 Toggle）等。

**表面交互的 UX 约束**：

| 约束编号 | 约束内容 |
|---|---|
| UI-1 | 表面滑块和 Toggle 的操作热区必须覆盖磁贴的完整表面面积，用户无需精准点击控件本身 |
| UI-2 | 任何表面交互操作产生的视觉反馈，必须在下一帧（ < 16ms）内在全域 UI 上体现，不得有任何延迟 |
| UI-3 | 正在进行表面交互操作的磁贴，其悬浮状态（Hover）和按压状态（Active）的物理动效必须暂停，避免与操作手势产生运动冲突 |
| UI-4 | 表面滑块在拖拽期间，必须显示当前值的实时数字气泡（Tooltip），以精确的数字反馈辅助模拟的物理感知 |

#### 4.4.2 下钻模式（Drill-Down Mode）

**定义**：用户点击磁贴（非拖拽操作），触发该磁贴的「下钻（Drill-Down）」动效，展开一个承载深度配置内容的子面板。

**下钻动效规范**：

| 阶段 | 动效描述 | 弹簧规格 |
|---|---|---|
| 点击触发 | 磁贴以 `scale(1.04)` 轻微「弹出」，表示响应用户点击 | `SPRING_SNAPPY` |
| 面板展开 | 子面板从磁贴边界向外扩张，以磁贴中心为原点执行 `scale(0.6) → scale(1.0)` 的放大入场，同时 `opacity: 0 → 1` | `SPRING_FLUID` |
| 背景遮蔽 | 控制中心网格中的其他磁贴透明度集体降低至 `0.30`，焦点集中于展开的子面板 | `SPRING_GENTLE` |
| 关闭 | 点击子面板外区域或子面板内的「返回」按钮，执行入场动效的镜像逆序退场 | `SPRING_SNAPPY` |

**下钻子面板的内容规范**：

- 子面板内部**禁止**出现任何传统的「保存」或「应用」按钮。所有深度配置的变更，必须实时生效（即改即存）。
- 敏感操作（如删除已知主机记录、变更 RASP 等级至 `SAFE` 或 `DEV`）除外，这类操作必须弹出原生确认对话框（Touch ID 或主密码验证）。
- 子面板的最大尺寸不得超过控制中心容器尺寸的 `85%`，确保子面板不会完全遮蔽背后的磁贴网格，维持用户的空间感知。

### 4.5 WWDC 2026 风格折射图标规范（Refractive Icon Spec）

#### 4.5.1 设计背景：为什么拒绝扁平 SVG

设置磁贴的图标是用户在视觉扫描过程中快速识别功能域的第一锚点。在 Liquid Glass 半透明基座上，扁平的单色 SVG 图标存在一个根本性的视觉缺陷：**当背景复杂或明亮时，单色图标会与背景「糊」在一起，丧失辨识度**。

MOOVIER SUPREME 引入 **WWDC 2026 风格的折射拟物图标（Refractive Skeuomorphic Icons）**，作为控制中心图标的强制规范。这类图标的特征是：

- **多层高光结构**：图标本体是一个三维形态的「玻璃实体」，具有顶面高光层、侧面渐变层和底部反射层，模拟光线从不同角度照射在玻璃或金属材质上的光学行为。
- **厚度感（Depth Perception）**：图标视觉上具有 Z 轴厚度，而非 SVG 的「贴纸」感。
- **折射色散（Chromatic Aberration）**：在高光边缘允许出现极轻微的彩色色差（蓝移/红移 1~2px），模拟真实玻璃棱镜的光学折射色散现象。

#### 4.5.2 折射图标的构成层次规范

每一个折射图标必须由以下视觉层次构成，无论是通过 SVG 渲染、Canvas 绘制还是 CSS 合成实现，都必须满足以下分层结构要求：

| 层次编号 | 层次名称 | 描述 |
|---|---|---|
| I-L1 | 底座阴影（Base Shadow） | 图标底部的短阴影，提供 Z 轴落地感。颜色为纯黑色，透明度 `15%~25%`，偏移 `Y: 2~3px`，模糊 `4px` |
| I-L2 | 主体材质层（Body Material） | 图标的主视觉体，材质为「黑玻璃」或对应功能的代表性材质（如安全图标为深金属色，AI 图标为深蓝折射色）|
| I-L3 | 环境遮蔽层（Ambient Occlusion） | 主体材质层的暗部，位于图标底部和侧面交接处，颜色比主体更深 `15%~30%`，模拟体积感光线的遮蔽区域 |
| I-L4 | 正面高光层（Specular Highlight） | 图标正面（朝向光源方向）的主高光，颜色为白色，透明度 `60%~80%`，形态为渐变色块（从顶部高亮渐变至底部消失）|
| I-L5 | 边缘折射线（Edge Refraction） | 图标轮廓内侧的极细 1px 高光线，模拟光线在玻璃边缘的全内反射，颜色为白色，透明度 `40%~60%` |
| I-L6 | 色散高光点（Chromatic Aberration） | 仅出现在图标最高点或最亮高光处，极小的彩色光点（蓝 + 红分离 1~2px），为可选层，不强制出现 |

#### 4.5.3 黑玻璃（Black Glass）材质规范

「黑玻璃」是控制中心磁贴的**默认材质规格**，其设计意图是：通过强烈的材质反差（深黑的玻璃体 vs 透明的 Liquid Glass 基座），确保设置磁贴在任何壁纸背景下都能被清晰识别，而不会「糊」进背景。

| 材质参数 | 规范值 |
|---|---|
| 基底色（深色模式） | `hsl(220, 10%, 6%)`（极深的冷调黑色，带少量蓝色偏移，区别于纯黑）|
| 基底色（浅色模式） | `hsl(220, 15%, 12%)`（深色磁贴在浅色模式下也维持深色，保持反差感）|
| 基底不透明度 | `0.85`（保留轻微透明度，允许底部 Liquid Glass 的光线隐约渗透，避免视觉完全封闭）|
| 内发光规格 | 遵循 §2.2.1 规范，但悬浮磁贴等级的参数：深色模式 `15%`，浅色模式 `60%` |
| 边缘阴影规格 | 遵循 §2.2.3 规范的悬浮磁贴等级参数 |
| 边缘圆角 | `--grid-unit * 4`（即 `16px`）。控制中心磁贴是全文档中**唯一允许使用非零圆角**的磁贴类型，以区别于应用主体的「建筑棱角」美学，提供 iOS/macOS 控制中心的亲切感。|

> **例外规则**：RASP 安全等级磁贴在激活「SAFE MODE」时，黑玻璃材质替换为「深红玻璃（`hsl(0, 60%, 8%)`）」；在激活「DEV MODE」时，替换为「深琥珀玻璃（`hsl(35, 70%, 8%)`）」，以提供强烈的视觉警示。

### 4.6 零延迟 WYSIWYG 实时反馈架构（Zero-Latency Live Preview Architecture）

#### 4.6.1 核心约束：绕过 React 渲染生命周期

React 的默认渲染模型是「状态变更 → 调度渲染 → Reconciliation → DOM Commit」——这条链路在高频输入（如拖拽滑块时每帧触发的 `onPointerMove`）场景下，即便经过充分的 `memo` 优化，其生命周期开销依然不可忽视，会导致视觉上可感知的「追帧滞后（Input Lag）」。

MOOVIER SUPREME 对所有「需要零延迟视觉反馈」的设置操作，规定使用**直接 CSS 变量注入（Direct CSS Variable Injection）**策略，完全绕过 React 渲染树，直接操作 DOM 根节点的 `style` 属性。

**零延迟操作的判定标准（以下任意一条满足即为零延迟操作）**：

1. 该操作的视觉反馈影响主界面的视觉外观（不仅限于控制中心内部）。
2. 该操作在高频输入（拖拽、滑动）场景下被触发。
3. 该操作的最终用户期望是「即改即见」（如主题色切换、透明度调整）。

#### 4.6.2 零延迟操作的双轨机制

零延迟操作必须同时在以下两条轨道上同步执行：

**轨道 A——即时视觉更新（CSS Variable Track）**

| 步骤 | 执行时机 | 操作 |
|---|---|---|
| A-1 | 用户输入事件触发的同一事件处理函数内（同步代码，不可异步延迟） | 直接调用 `document.documentElement.style.setProperty('--css-var-name', newValue)`，将变更立即写入 `:root` 的内联样式 |
| A-2 | 浏览器渲染引擎 | 检测到 CSS 变量变更，在下一帧（< 16ms）自动重新计算所有引用了该变量的 CSS 属性，触发视觉更新 |

> **A 轨道的绝对约束**：A-1 步骤的执行路径上，**禁止**出现任何 `setState`、`dispatch`、`useReducer` 调用，也**禁止**出现任何会触发 React 组件重渲染的操作。这条轨道的全部逻辑，必须运行在 React 渲染树之外。

**轨道 B——持久化状态同步（Zustand Track）**

| 步骤 | 执行时机 | 操作 |
|---|---|---|
| B-1 | 用户输入事件结束时（`onPointerUp`、`onChange` 最终值稳定） | 调用 Zustand Store 的 `set` 方法，将最终值写入全局状态树 |
| B-2 | Zustand Middleware | 检测到 Store 变更，触发持久化操作（写入 `localStorage` 或通过 IPC 写入主进程配置） |
| B-3 | 主进程（可选，针对需要持久化到磁盘的配置） | 接收配置更新 IPC，写入 `app-config.json` |

> **B 轨道的约束**：B-1 步骤必须是**防抖（Debounced）**的，防抖延迟 `300ms`。这意味着用户拖拽滑块期间，B 轨道静默等待；只有在用户停止拖拽超过 300ms 后，才触发一次 Zustand 状态同步。这确保了滑块拖拽期间的持久化操作不会产生大量无意义的磁盘 I/O。

#### 4.6.3 零延迟操作的 CSS 变量映射表

以下为标准设置操作与 CSS 变量的完整映射关系，所有实现必须严格遵循：

| 设置磁贴 | 用户操作 | A 轨道写入的 CSS 变量 | B 轨道写入的 Zustand Key |
|---|---|---|---|
| 透明度控制 | 拖拽滑块 | `--glass-transparency` | `settings.glassTransparency` |
| 终端背景遮罩 | 拖拽滑块 | `--terminal-isolation-opacity` | `settings.terminalIsolationOpacity` |
| 主题色板 | 点击色块 | `--primary-color` | `settings.primaryColor` |
| 字体大小 | 拖拽滑块 | `--terminal-font-size` | `settings.terminalFontSize` |
| 深色/浅色 | 点击切换 | `--color-scheme`（触发 `:root[data-theme]` 切换） | `settings.colorScheme` |
| 毛玻璃开关 | Toggle | `--glass-blur-radius`（`0px` = 关闭）| `settings.glassEnabled` |

#### 4.6.4 「即改即存」原则（Auto-Persist on Change）

控制中心的所有设置操作，遵循以下统一的持久化原则：

- **禁止「保存」按钮**：控制中心内任何位置，包括下钻子面板，绝对禁止出现「保存」、「确认」、「应用」等需要用户二次确认才能生效的按钮。
- **禁止「放弃更改」逻辑**：关闭控制中心（点击遮罩或按 `Esc`）不会「回滚」任何已做的更改，所有 B 轨道已写入的 Zustand 状态和持久化数据均保留。
- **例外操作（需确认）**：以下操作因其不可逆性或高权限性质，例外要求原生确认对话框：撤销已知主机信任记录、卸载插件、切换 RASP 等级至 `SAFE` 或 `DEV`、删除 AI 密钥。

## 第 5 章：跨平台一致性与优雅降级策略 (Cross-Platform Consistency & Graceful Degradation)

### 5.1 设计哲学：降级不是妥协，是有尊严的适应（Degradation with Dignity）

MOOVIER SUPREME 的视觉宪法以 macOS 14+ 上的 Liquid Glass 原生材质为最高表现形态构建。然而，GETSSH 的用户群横跨 macOS、Windows 11、Windows 10 和 Linux 四个平台，硬件配置从高性能工作站到资源受限的云端 VPS 客户端不等。

一个脆弱的系统，会在非理想环境下崩溃、渲染异常、或呈现出破损的视觉状态——白色背景透出、毛玻璃失效后的灰色残骸、不同步的布局偏移。MOOVIER SUPREME 绝不允许这类情况发生。

**降级的根本原则**：

> 在任何平台、任何硬件条件下，GETSSH 3.0 的界面**始终**应当呈现出「这是一款精心设计的专业工具」的观感——即便在最严苛的降级状态下。降级的目标不是「还能用」，而是「在当前条件下的最佳视觉表达」。

为实现这一目标，本章规定了三套互相独立、可组合叠加的降级与适配防御机制：材质表现力降级金字塔、动画与渲染算力降级策略、窗体控制组件跨平台对齐规范。

---

### 5.2 材质表现力降级金字塔（The Material Degradation Pyramid）

#### 5.2.1 金字塔总体结构

降级金字塔共分三个梯队，从上到下，材质的原生渲染能力依次递减，但视觉设计的尊严防线始终维持。主进程在应用启动时必须执行一次**平台环境探测（Platform Capability Detection）**，并将探测结果通过 IPC 注入渲染进程，由渲染进程据此激活对应梯队的 CSS 变量组和类名开关。

```
┌─────────────────────────────────────────────────────┐
│  第一梯队：macOS 14+（Sequoia 及以上）              │
│  原生 Vibrancy + Liquid Glass 全特性                 │
├─────────────────────────────────────────────────────┤
│  第二梯队：Windows 11（Build 22000+）                │
│  Mica / Acrylic 原生材质 + 黑玻璃强化补偿           │
├─────────────────────────────────────────────────────┤
│  第三梯队：Windows 10 / Linux / 硬件加速关闭        │
│  实色深色系底座 + 微亮磁贴 + 绝对网格秩序维持       │
└─────────────────────────────────────────────────────┘
```

#### 5.2.2 第一梯队：macOS 14+（Liquid Glass 全特性模式）

**适用条件**：`process.platform === 'darwin'` 且 `os.release()` 所对应的 macOS 版本 ≥ 14.0（Sonoma）。

**主进程窗口配置规范**：

| 参数 | 规范值 |
|---|---|
| `vibrancy` | `'under-window'`（覆盖整个窗口下方的系统内容）|
| `visualEffectState` | `'active'`（始终激活视觉效果，不随窗口焦点状态改变）|
| `backgroundColor` | `'#00000000'`（完全透明，将着色权完全交给 CSS 变量层）|
| `titleBarStyle` | `'hiddenInset'`（隐藏默认标题栏，红绿灯按钮保留并嵌入自定义布局）|
| `trafficLightPosition` | 遵循 §5.4.2 中定义的绝对网格对齐规范 |

**渲染进程 CSS 根节点激活标志**：`<html data-platform="macos-tier1">`

**第一梯队专属激活特性清单**：

| 特性 | 激活状态 |
|---|---|
| `--glass-transparency` 变量生效 | ✅ 全量生效，允许用户从 0.0 调至 1.0 |
| OS 原生窗口 Vibrancy 材质 | ✅ 激活，提供最纯粹的 Liquid Glass 底座 |
| 全局 `backdrop-filter` 使用权 | ✅ 允许在 CC Container 的 Scrim 层使用 |
| 磁贴三层复合阴影 | ✅ 全量渲染（见 §2.2.3）|
| 弹簧物理动效 | ✅ 全特性运行，无帧率限制（受设备实际性能自然约束）|
| 多窗口撕裂与空壳窗口池化 | ✅ 激活（见第 3 章）|
| 折射图标六层结构 | ✅ 全层渲染（见 §4.5.2）|

#### 5.2.3 第二梯队：Windows 11（Mica/Acrylic 材质补偿模式）

**适用条件**：`process.platform === 'win32'` 且 Windows 版本 Build ≥ 22000（Windows 11 正式版）。

**Windows 11 上的材质现实**：Windows 的 Mica 材质（`backgroundMaterial: 'mica'`）从桌面壁纸采样并进行高斯模糊，而非从窗口下方的实时内容采样。这意味着它在视觉效果上与 macOS Vibrancy 存在本质差异：Mica 是「壁纸的模糊回声」，而 Vibrancy 是「窗口下方内容的实时折射」。设计规范必须正视这一差距并采取专项补偿策略。

**主进程窗口配置规范**：

| 参数 | 规范值 | 说明 |
|---|---|---|
| `backgroundMaterial` | 优先尝试 `'mica'`，若失败则回退至 `'acrylic'` | Mica 优先，视觉更沉稳；Acrylic 模糊更强，作为备选 |
| `backgroundColor` | `'#00000000'`（完全透明）| 与第一梯队一致 |
| `titleBarStyle` | `'hidden'`（完全隐藏，使用自定义 Frameless 窗口）| Win11 不支持 `hiddenInset`，必须完全自定义标题栏区域 |

**第二梯队专属适配策略——黑玻璃层级强化补偿（Black Glass Layering Compensation）**：

由于 Mica 材质的层次感弱于原生 Vibrancy，磁贴与背景之间的视觉剥离感需要通过**加深黑玻璃参数**来主动补偿。

| 参数 | 第一梯队值 | 第二梯队调整值 | 说明 |
|---|---|---|---|
| 磁贴表面着色层（深色模式）| `rgba(0, 0, 0, 0.20)` | `rgba(0, 0, 0, 0.30)` | 磁贴着色加深 10%，增强与 Mica 背景的对比 |
| 复合阴影第 1 层透明度（主磁贴）| `0.30` | `0.40` | 近场轮廓阴影加深，强化边界感 |
| 复合阴影第 2 层透明度（主磁贴）| `0.15` | `0.22` | 中场阴影加深，增强浮起感 |
| 内发光透明度（主磁贴深色模式）| `10%` | `14%` | 边缘高光略微增强，对抗 Mica 背景噪点的视觉干扰 |
| 黑玻璃控制中心磁贴基底不透明度 | `0.85` | `0.92` | 控制中心磁贴更实，与 Mica 背景区分度更高 |

**噪点纹理叠加规范（Noise Texture Overlay）**：

Windows 11 的 Acrylic 材质因其高频噪点颗粒，在视觉上与磁贴材质存在「纹理冲突」（磁贴的磨砂质感与 Acrylic 的噪点质感叠加后产生视觉杂乱）。当检测到系统使用 Acrylic 回退时，渲染进程须在 L2 应用着色层与 L3 网格骨架层之间，注入一个额外的**噪点抑制遮罩层（Noise Suppression Scrim）**：

| 参数 | 规范值 |
|---|---|
| 层级 | L2.5（在 L2 着色层之上，L3 网格层之下）|
| 颜色 | `rgba(13, 14, 18, 0.25)`（与第三梯队实色背景同色系，轻度压制底部噪点）|
| 作用 | 轻度压制 Acrylic 噪点颗粒的视觉穿透强度，不完全消除（保留材质感），仅降低其对磁贴清晰度的干扰 |

**渲染进程 CSS 根节点激活标志**：`<html data-platform="windows-tier2">`

#### 5.2.4 第三梯队：Windows 10 / Linux / 硬件加速关闭（实色尊严模式）

**适用条件**：以下任意一条满足即进入第三梯队：
- `process.platform === 'win32'` 且 Windows 版本 Build < 22000（Windows 10 及以下）
- `process.platform === 'linux'`
- Electron 的 `app.getGPUInfo()` 返回结果中，硬件加速状态为已禁用或不可用
- 用户在设置中手动开启「性能模式（关闭所有视觉效果）」

**核心设计决策：以实色系的绝对秩序对抗材质缺失**

在没有任何原生窗口材质支撑的情况下，界面的质感来源只剩下两个：**颜色的精确控制**和**布局的绝对秩序**。第三梯队的设计就是将这两个维度做到极致。

**主进程窗口配置规范**：

| 参数 | 规范值 |
|---|---|
| `backgroundColor` | `'#0d0e12'`（与实色背景底座完全一致，防止窗口初始化时的白色闪烁）|
| `titleBarStyle` | `'hidden'`（同第二梯队，完全自定义）|
| 所有 vibrancy / backgroundMaterial 相关配置 | 不设置（在不支持的系统上设置这些参数会导致渲染异常）|

**实色背景系（Solid-Color Palette for Tier-3）规范**：

| 色彩角色 | CSS 变量 | HSL 值 | 说明 |
|---|---|---|---|
| 应用底座色（Root Background） | `--bg-root` | `hsl(228, 12%, 7%)`（`#0d0e12`）| 近黑色冷调深蓝，为整个窗口提供沉稳底色 |
| 主磁贴表面色（Primary Tile Surface） | `--bg-tile-primary` | `hsl(228, 10%, 11%)`（`#161822`）| 比底座浅约 4 个亮度单位，创造可感知的层次感 |
| 次磁贴表面色（Secondary Tile Surface） | `--bg-tile-secondary` | `hsl(228, 9%, 13%)`（`#1b1d26`）| 监控面板等次磁贴的表面色 |
| 侧边栏表面色（Sidebar Surface） | `--bg-sidebar` | `hsl(228, 13%, 9%)`（`#111318`）| 比底座略深，与主内容区形成侧向层次差 |
| 状态栏表面色（Statusbar Surface） | `--bg-statusbar` | `hsl(228, 14%, 8%)`（`#0f1015`）| 接近底座色，状态栏保持低调 |
| 磁贴边缘线（Tile Border） | `--border-tile` | `rgba(255, 255, 255, 0.06)` | 极细的半透明白色边缘线，替代第一/二梯队的 1px 内发光 |

> **色阶设计原则**：以上实色调色板以 `4` 个 HSL 亮度（Lightness）单位为一个色阶步长，完全对应第 1 章定义的 `--grid-unit` 精神——数值上的绝对秩序在色彩体系中的映射体现。视觉层次来自精确的数值控制，而非随意的颜色选择。

**第三梯队的特性降级清单**：

| 特性 | 降级状态 | 降级方案 |
|---|---|---|
| OS 原生窗口材质 | ❌ 不可用 | 使用 `--bg-root` 实色替代 |
| 全局 `backdrop-filter` | ❌ 禁止使用（性能开销无法接受）| CC Container Scrim 层降级为半透明实色 `rgba(0, 0, 0, 0.70)` |
| 磁贴三层复合阴影 | ⚠️ 降级 | 仅保留第 1 层近场轮廓阴影，第 2、3 层省略 |
| 1px 内发光（Inner Glow）| ⚠️ 降级 | 替换为 `--border-tile` 单侧边缘线，视觉语义不变，实现方式简化 |
| `--glass-transparency` 滑块 | ❌ 锁定 | 强制设为 `0.0`（完全着色），透明度控制磁贴在控制中心内隐藏 |
| 弹簧物理动效 | ⚠️ 降级 | 见 §5.3 中的动效降级规范 |
| 多窗口撕裂（Tear-off）| ❌ 禁用 | 侧边栏「在新窗口中打开」功能通过标准 `new BrowserWindow()` 实现，不使用空壳窗口池 |

**渲染进程 CSS 根节点激活标志**：`<html data-platform="tier3-solid">`

### 5.3 动画与渲染算力降级策略（Performance & Animation Scaling）

#### 5.3.1 设备性能分级探测（Device Capability Profiling）

应用启动时，主进程须执行一次**设备性能画像（Device Capability Profile）**，综合以下信息生成一个性能等级枚举值（`PERF_LEVEL`），通过 IPC 注入渲染进程：

| 探测维度 | 探测方法 | 说明 |
|---|---|---|
| 逻辑 CPU 核心数 | `os.cpus().length` | < 4 核视为低性能标志 |
| 系统可用内存 | `os.freemem()` / `os.totalmem()` | 可用内存占比 < 20% 视为内存压力状态 |
| GPU 硬件加速可用性 | `app.getGPUFeatureStatus()` | 任何关键 GPU 特性被软件降级，即为加速不可用 |
| 操作系统省电模式 | macOS: `powerMonitor.isOnBatteryPower()` + `powerMonitor.getSystemIdleState()`；Windows: WMI 查询电源计划 | 省电模式激活时主动触发动效降级 |
| 用户手动偏好 | 设置 `settings.performanceMode` | 用户在控制中心手动开启「性能模式」，具有最高优先级 |

**性能等级枚举定义**：

| 等级 | 名称 | 触发条件 |
|---|---|---|
| `PERF_HIGH` | 高性能模式 | 默认状态，所有探测维度均在正常范围内 |
| `PERF_BALANCED` | 均衡模式 | CPU 核心数 < 4，或系统内存压力较高，或使用电池但非省电模式 |
| `PERF_SAVER` | 省电/低端模式 | 省电模式激活，或 GPU 加速不可用，或用户手动开启 |

#### 5.3.2 弹簧动效的梯度降级规范

Framer Motion 的弹簧物理动效是 MOOVIER SUPREME 的核心视觉语言，但其逐帧物理计算的计算成本在低端设备上不可忽视。以下为三个性能等级对应的动效策略：

**`PERF_HIGH`（高性能模式）**：

- 所有弹簧动效全量运行，参数遵循 §2.3.1 中定义的弹簧参数基准库，不施加任何额外约束。
- 帧率不锁定，交由设备和浏览器渲染引擎自然决定（通常为显示器刷新率，60Hz / 120Hz）。

**`PERF_BALANCED`（均衡模式）**：

- 弹簧动效保留，但参数收紧：
  - 所有弹簧的 `stiffness`（硬度）参数 × `1.5`（动效更快收敛，减少物理计算帧数）
  - 所有弹簧的 `mass`（质量）参数统一降为 `0.8`（动效更轻盈，超调幅度减小）
- 以下低优先级动效在均衡模式下停用：
  - 磁贴拖拽期间的实时角速度旋转（`Tilt on Drag`，见 §3.2.2）
  - 复合阴影随拖拽速度的实时更新
  - 弹簧超调（Overshoot）行为（通过 `restDelta` 参数收紧至 `0.5` 实现）
- 帧率软限制为 `60fps`（由 `requestAnimationFrame` 的自然节拍保证，不强制节流）。

**`PERF_SAVER`（省电/低端模式）**：

- 弹簧物理动效**完全停用**，Framer Motion 动效统一降级为 CSS `transition` 实现的**线性淡入淡出（Linear Fade Transitions）**。
- 线性过渡参数规范：

| 动效类型 | 过渡时长 | 缓动函数 | 说明 |
|---|---|---|---|
| 状态切换（Toggle、Tab） | `120ms` | `ease-out` | 快速响应，保持应用流畅感 |
| 面板/覆层入场 | `180ms` | `ease-out` | 轻微入场缓动，避免突兀 |
| 面板/覆层退场 | `120ms` | `ease-in` | 退场更快，符合「收起」的直觉 |
| 悬浮状态（Hover） | `80ms` | `ease-out` | 极快的 Hover 响应，不拖泥带水 |
| 所有按压/释放动效 | 停用（即时切换）| — | 按压反馈改为即时颜色变化，无过渡 |

- 帧率强制锁定至 `30fps`：通过在渲染进程中以 `setInterval(requestAnimationFrame, 33)` 模式对动效循环进行节流，将渲染请求频率压制到约 30fps，全面释放 CPU 算力。

> **帧率节流的精确实现约束**：节流逻辑必须作用于**动效刷新循环**，而非整个渲染进程的帧率。xterm.js 的终端输出渲染（PTY 字符输出到 Canvas 的绘制）**不受此节流影响**，必须始终以最高优先级、不受限制的帧率运行，确保终端输出的实时性不因动效降级而受损。

#### 5.3.3 xterm.js 渲染层降级规范

**适用条件**：以下任意一条满足，xterm.js 必须降级其渲染器：
- `app.getGPUFeatureStatus()` 显示 WebGL 或 WebGPU 渲染不可用
- 用户显卡驱动程序版本过旧，Canvas WebGL 上下文创建失败
- `PERF_SAVER` 模式激活

**降级路径（严格单向）**：

```
WebGL/WebGPU Canvas 渲染器（默认，硬件加速）
          ↓ 检测到硬件加速不可用时（单次自动降级，不可逆）
Canvas 2D 渲染器（软件加速，CPU 渲染）
          ↓ 检测到 Canvas 2D 也不可用时（极端情况）
DOM 渲染器（纯 HTML DOM 节点渲染，最大兼容性）
```

**降级行为约束规范**：

| 约束编号 | 约束内容 |
|---|---|
| R-1 | 降级切换必须**无感发生**——即用户不会看到任何空白闪烁、黑屏或渲染重启提示。xterm.js 实例在内部完成渲染器切换，外部 DOM 容器不重新创建 |
| R-2 | 降级切换必须在**单次检测失败后立即触发**，不允许重试多次或等待超时后再触发，防止渲染死锁导致整个应用界面挂起 |
| R-3 | 降级状态信息必须通过 IPC 上报主进程，主进程将其记录至日志，并在状态栏磁贴上显示一个小型的降级状态徽章（如「⚡ 软件渲染」），告知用户当前的渲染状态，供故障排查参考 |
| R-4 | PTY 的 SharedArrayBuffer 数据流（见 §3.4）在渲染器降级后**保持不变**。渲染器的切换只是「谁来读取 SAB 并绘制」的变化，与数据层完全解耦 |

### 5.4 窗体控制组件的跨平台绝对对齐（Window Controls Alignment）

#### 5.4.1 设计背景：窗体控制与绝对网格的冲突与融合

第 1 章建立的绝对网格体系要求应用内所有 UI 元素与 `4px` 网格精确对齐。窗体控制按钮（macOS 红绿灯、Windows 标题栏三大件）是操作系统原生注入的 UI 元素，其位置、尺寸和交互行为由 OS 决定，并非 CSS 布局引擎管辖——这是绝对网格体系面临的最复杂的对齐挑战。

**核心目标**：无论在哪个平台上，窗体控制按钮区域必须：

1. 与第 1 章定义的侧边栏磁贴的顶部边缘**精确水平对齐**（同一水平基准线）。
2. 其外边缘留白与侧边栏磁贴的边缘留白**精确匹配**（同一垂直基准线，遵循边缘抵达规则 E-1）。
3. 控制按钮区域本身的视觉材质，与相邻磁贴**统一**，不得出现原生 OS 标题栏的默认灰色或白色背景残留。

#### 5.4.2 macOS 红绿灯对齐规范

**适用梯队**：第一梯队（macOS 14+）

macOS 在使用 `titleBarStyle: 'hiddenInset'` 时，允许通过 `trafficLightPosition` 参数精确控制红绿灯按钮的位置（相对于窗口左上角的 `{ x, y }` 偏移量）。

**像素级对齐规范**：

| 参数 | 规范值 | 说明 |
|---|---|---|
| `trafficLightPosition.x` | `--grid-unit * 4`（`16px`）| 红绿灯左边缘距窗口左侧 16px，与侧边栏磁贴的内左间距对齐 |
| `trafficLightPosition.y` | `(--sidebar-header-height - 12px) / 2`（垂直居中于侧边栏顶部 Header 区域）| 红绿灯在侧边栏 Header 行内垂直居中 |
| 红绿灯可见区高度占用 | `--sidebar-header-height`（与侧边栏顶部 Header 区域等高，不扩展额外区域）| 确保拖拽区域不侵入主内容区 |
| `--sidebar-header-height` 定义 | `--grid-unit * 10`（`40px`）| 见第 1 章绝对网格规范 |

**macOS 拖拽区（Drag Region）规范**：

| 参数 | 规范值 |
|---|---|
| 拖拽区（`-webkit-app-region: drag`）范围 | 仅限侧边栏顶部 Header 区域（宽度 = 侧边栏宽度，高度 = `--sidebar-header-height`）|
| 主内容区（终端、面板）| 强制设置 `-webkit-app-region: no-drag`，防止终端文字选取操作被误识别为拖拽 |
| 红绿灯按钮自身区域 | 浏览器自动排除（`no-drag`），无需显式设置 |

#### 5.4.3 Windows / Linux 自定义标题栏规范

**适用梯队**：第二梯队（Windows 11）、第三梯队（Windows 10 / Linux）

在 `titleBarStyle: 'hidden'`（完全 Frameless）模式下，操作系统不注入任何原生窗口控制按钮。GETSSH 3.0 必须在渲染进程中自行实现**自定义标题栏控制块（Custom Window Controls Bar）**。

**自定义标题栏控制块结构规范**：

| 组成元素 | 说明 |
|---|---|
| 最小化按钮（Minimize）| 调用 `ipcRenderer.send('window:minimize')`，主进程执行 `win.minimize()` |
| 最大化/还原按钮（Maximize/Restore）| 调用 `win.maximize()` / `win.unmaximize()`，按钮图标根据当前窗口状态动态切换 |
| 关闭按钮（Close）| 调用 `win.close()`，关闭前触发应用级的「会话保护」询问逻辑（若有未保存 Session）|
| 拖拽区（Drag Region）| 整个标题栏控制块容器（不含按钮本身）设置为 `-webkit-app-region: drag` |

**控制块的绝对网格对齐规范**：

| 参数 | 规范值 |
|---|---|
| 控制块高度 | `--sidebar-header-height`（`40px`），与侧边栏顶部 Header 区域等高 |
| 控制块宽度 | `auto`（由三个按钮的尺寸自然决定）|
| 控制块位置 | 位于窗口右上角，`position: fixed; top: 0; right: 0` |
| 按钮内部间距 | `--grid-unit * 2`（`8px`）|
| 按钮点击热区 | `40px × 40px`（即 `--sidebar-header-height` × `--sidebar-header-height`，确保触达友好）|
| 控制块右边距 | `0`（按钮组右侧与窗口右边缘完全贴合，遵循边缘抵达规则 E-1）|

**关键约束——毛玻璃剔除处理（Glass Exclusion for Win Controls）**：

在第二梯队（Windows 11 / Mica 模式）下，自定义标题栏控制块区域必须执行**毛玻璃剔除**处理，确保按钮区域的视觉材质与整体磁贴系统统一，而不是显示出 Mica 材质在该区域的原始渲染。

| 处理方案 | 规范值 | 说明 |
|---|---|---|
| 控制块背景色 | `rgba(13, 14, 18, 0.80)`（与第三梯队底座色一致，高不透明度）| 用实色遮蔽 Mica 在按钮区域的渲染，使按钮区域在视觉上更接近第三梯队的实色磁贴，而非透过 Mica 展示桌面壁纸 |
| 控制块材质等级 | 辅助磁贴（Auxiliary Tile）等级，遵循 §2.2 规范中辅助磁贴的所有光影参数 | 确保按钮区域在整个磁贴视觉语言体系中具有一致的物理质感 |
| 按钮 Hover / Active 状态 | Hover: 背景色 `rgba(255, 255, 255, 0.08)`；Active: 背景色 `rgba(255, 255, 255, 0.14)` | 关闭按钮的 Hover 状态例外：使用 `rgba(196, 43, 28, 0.85)`（Windows 原生红色），尊重用户的平台肌肉记忆 |
| 关闭按钮的 Active 状态 | `rgba(196, 43, 28, 1.0)` | 完全不透明红色，提供强烈的操作确认感 |

#### 5.4.4 拖拽区的跨平台统一规范

无论在哪个平台，以下拖拽区规范必须统一执行，不得有平台特例：

| 规则 | 规范内容 |
|---|---|
| 拖拽区范围 | 仅限侧边栏顶部 Header 区域（macOS）或自定义标题栏控制块的空白区域（Windows/Linux）。**任何**终端磁贴、监控面板磁贴、WelcomePane 区域均不得设置为可拖拽区 |
| 双击行为 | 双击拖拽区执行「最大化/还原」切换（macOS 通过 Electron API 控制，Windows 通过自定义按钮逻辑实现）|
| 右键行为 | 禁止。拖拽区的右键菜单（如果 OS 有默认行为）必须被 `event.preventDefault()` 阻止 |
| 拖拽区与弹簧动效的冲突 | 拖拽区内的任何子元素，若同时被设置为「局部拖拽（Local Drag）」磁贴，其拖拽事件必须通过 `e.stopPropagation()` 防止冒泡至窗口层拖拽逻辑，避免用户拖动内部元素时意外移动整个窗口 |

### 5.5 降级状态的全局感知与运行时切换（Global Degradation Awareness & Runtime Switching）

#### 5.5.1 降级状态注入机制

主进程在完成**平台环境探测**和**设备性能画像**后，须通过以下机制将降级状态注入渲染进程，确保整个渲染层的 CSS 和逻辑层都能感知当前的运行环境：

| 注入维度 | 注入方式 | 渲染进程接收位置 |
|---|---|---|
| 平台梯队 | IPC 启动参数，设置 `<html data-platform="...">` 属性 | CSS 选择器 `[data-platform="macos-tier1"]` 等 |
| 性能等级 | IPC 启动参数，设置 `<html data-perf="high|balanced|saver">` 属性 | CSS / JS 均可读取 |
| 省电模式变化 | `powerMonitor` 事件监听，实时通过 IPC 推送至渲染进程 | Zustand `system.perfLevel` 状态更新，触发动效策略实时切换 |
| GPU 状态变化 | 目前 Electron 不支持 GPU 状态的运行时推送，仅在启动时探测一次 | 启动时固定，不支持热切换 |

#### 5.5.2 降级状态的运行时热切换（省电模式）

省电模式是唯一一个在运行时可能动态变化的降级维度（用户可能在使用过程中将笔记本接入电源或拔出电源）。当 `powerMonitor` 事件触发省电模式状态变化时，渲染进程须执行以下实时切换序列：

| 步骤 | 操作 | 动效 |
|---|---|---|
| S-1 | 渲染进程接收 `system:perf-level-changed` IPC 事件，更新 Zustand `system.perfLevel` 状态 | — |
| S-2 | 将 `<html data-perf="...">` 属性更新为新的性能等级 | — |
| S-3 | 若从 `PERF_HIGH` / `PERF_BALANCED` 切换至 `PERF_SAVER`：停用所有正在运行的弹簧动效实例，切换为 CSS Transition 策略 | 切换本身不执行任何动效（直接切换，防止在低性能状态下执行切换动效造成卡顿）|
| S-4 | 若从 `PERF_SAVER` 切换至 `PERF_HIGH` / `PERF_BALANCED`：重新激活弹簧动效策略 | 以 `SPRING_GENTLE` 弹簧规格执行一次全域透明度「呼吸」动效（`opacity: 0.95 → 1.0`），作为「性能提升已激活」的视觉确认信号 |
| S-5 | 在状态栏磁贴上更新性能模式指示徽章 | 徽章切换采用 CSS Transition（当前处于 SAVER 模式时）或 `SPRING_SNAPPY`（当前处于高性能模式时）|

### 5.6 宪法完结条款：五章守恒原则

本文档五个章节，各自负责一个独立的工程域，但共享同一套设计哲学根系。在任何实现层面产生规范冲突时，冲突解决优先级遵循以下守恒原则：

| 优先级 | 原则名称 | 定义 |
|---|---|---|
| P1（最高）| **可用性守恒** | 任何视觉或交互设计决策，不得损害 P0 级文字可读性（WCAG AAA，见 §2.4）。可用性高于一切美学追求。 |
| P2 | **数据完整性守恒** | PTY 状态流的单一权威来源（Rust 引擎 + SAB，见 §3.4）不得被任何视觉层的决策破坏。渲染器的降级或切换，永远不触碰数据层。 |
| P3 | **绝对网格守恒** | `4px` 网格单元和边缘抵达规则（E-1、E-2、E-3，见 §1.3）是布局的物理法律，任何平台适配（第 5 章）均不得破坏网格对齐精度。 |
| P4 | **物理质感守恒** | 在当前平台和设备能力允许的最高梯队内，弹簧物理动效和磁贴光影规范必须被完整执行。不允许在高性能平台上主动省略规定的物理效果。 |
| P5（最低）| **美学一致性守恒** | 在 P1~P4 均满足的前提下，所有设计决策优先选择与已有规范视觉语言最接近的方案，避免产生孤立的、风格突兀的 UI 区域。 |
