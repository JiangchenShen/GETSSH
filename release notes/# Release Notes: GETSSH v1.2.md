# Release Notes: GETSSH v1.2.0 (The Foundation Update)

距离 GETSSH v1.0 的首次发布仅过去短短 72 小时，开发团队在这个周期内完成了多达 **115 次核心代码提交**。如果说 v1.0 向开发者社区展示了“一个轻量级、安全的现代 SSH 客户端应该是什么样”，那么 **v1.2.0 则是一场不折不扣的底层架构重塑**。

本次更新没有盲目堆砌表层功能，而是将全部精力倾注于**极限瘦身、跨平台编译、核心状态重构以及致命安全漏洞的封堵**。我们明确宣布：GETSSH 目前的 1.x 系列版本其唯一核心使命就是“打造坚不可摧的底层地基”，为即将到来的革命性功能铺路。**完整的、可供第三方开发者接入的 Plugin SDK，已正式定档于 GETSSH v2.0 推出。**

以下是 v1.2.0 版本的详细技术剖析与更新日志。

---

## 📉 极限瘦身解密：从 460MB 到 90MB 的跨越

在 v1.0 版本中，Windows 与 macOS 的安装包落地体积高达 400MB 甚至 460MB，这对于一个标榜“轻量”的终端工具而言是不可接受的。在 v1.2.0 中，我们将全平台安装包的体积**极其严苛地压榨到了 90MB 以下（Windows 仅 73MB）**。

**我们是如何做到的？**
1. **深度依赖剔除 (Dependency Pruning)与 Tree-Shaking**：v1.0 的打包产物中意外混入了大量未使用的 Node.js 开发期依赖 (devDependencies) 以及冗余的 Electron 预编译二进制文件。本次更新中，我们重写了打包流水线，利用更为激进的模块打包策略，确保只有运行时绝对需要的代码被包含进最终的 ASAR 归档中。
2. **原生模块针对性重编译**：针对 `cpu-features` 等底层 C++ 原生模块，我们摒弃了臃肿的通用编译方案，在 CI/CD 流程中为目标平台的特定架构（arm64/x64）执行了精准的交叉编译（Cross-compilation），剔除了所有非必要的平台兼容层代码。
3. **资源深度压缩**：重构了静态资源的加载与打包逻辑，使得应用程序在保持极速启动的同时，磁盘占用率实现了 **80% 的断崖式下降**。

---

## 🌍 全平台制霸：新增 Intel Mac 与 Linux 官方支持

在 v1.0 时代，我们仅提供了 Windows (x64) 和 macOS (Apple Silicon / ARM64) 的预编译包，Linux 与老款 Mac 用户只能被迫拉取源码自行编译。v1.2.0 彻底终结了这一痛点，实现了真正的全平台官方分发：

* **新增 macOS Intel (x64) 原生支持**：为庞大的老款 MacBook 及 iMac 用户群体提供了专属优化的 `.dmg` 安装包，底层 SSH 连接与加密模块针对 x64 架构进行了重新适配。
* **新增 Linux 通用版 (AppImage)**：为了规避 Linux 发行版碎片化带来的“依赖地狱 (Dependency Hell)”，我们选用了 `AppImage` 格式。用户无需关注系统的 glibc 版本或包管理器差异，下载后赋予执行权限即可直接运行，完美契合 GETSSH 开箱即用的哲学。

## 🛡️ 对比 v1.0：严重安全漏洞修复与 I/O 性能跃迁

v1.0 版本中的 `PluginManager` 模块在设计初期遗留了部分高危缺陷及性能瓶颈，v1.2.0 对此进行了精准修复：

1. **彻底修复 Zip Slip 与路径穿越 (Path Traversal) 漏洞**：
   * *问题重现*：v1.0 的插件解压逻辑未对压缩包内的文件路径进行严格的合法性校验。恶意构造的插件包可以通过 `../../` 类的相对路径逃逸出安装目录，执行任意文件覆盖（Arbitrary File Write），甚至篡改系统核心文件。
   * *解决方案*：引入了极其严密的路径边界断言机制。所有插件的解压与读写动作被严格限制在预设的绝对路径沙盒内，彻底封死了通过第三方插件攻击宿主机操作系统的可能。
2. **根除主进程阻塞 (ANR 防御)**：
   * *问题重现*：v1.0 在处理插件安装/卸载时，Electron 主进程（Main Process）使用了 `fs.readFileSync` 等同步阻塞式 I/O 方法。这会导致在读写大量文件时，整个事件循环（Event Loop）被卡死，前端 UI 出现严重的假死现象（Application Not Responding）。
   * *解决方案*：将所有同步文件流操作全盘重构为 `fs.promises` 异步架构。配合严谨的 `async/await` 异常捕获机制，目前的 GETSSH 即使在后台处理海量 I/O 操作，终端输入与 UI 渲染依然保持电竞级的丝滑响应。

---

## 🏗️ 全新引入：状态引擎与安全沙盒

对比 v1.0 较为松散的前端架构，v1.2.0 引入了多项决定未来上限的核心基建：

* **Zustand 全局状态统一**：彻底清除了入口文件（`App.tsx`）中臃肿且易冲突的 React `useState` 逻辑。全面迁移至 Zustand 状态树（涵盖 `useAppStore`, `useSessionStore`, `usePluginStore`）。这不仅消除了配置不同步的数据漂移风险，更使得业务逻辑与 UI 渲染彻底解耦。
* **100% 测试覆盖的业务中枢**：针对全新的 Store 模块，我们编写了全覆盖的 Vitest 单元测试。无论是复杂的标签页管理（Tab Management）还是主题同步，均在持续集成中受到严格的边界测试保护。
* **引入 Iframe 沙盒边界 (`initPluginBridge`)**：为未来的前端插件化做好了物理隔离。基于 Context Bridge 和严格的 Iframe 策略，插件代码在完全受限的上下文中运行，无法越权访问宿主程序的 DOM 或内存状态。
* **全新 SplitPane 布局引擎**：重写了底层的视窗管理器，不仅优化了现有的终端渲染区域，更为未来高级的分屏操作、侧边栏面板拓展预留了标准化的挂载点。

## 🚀 展望 v2.0：插件生态的正式启航

我们要向一直关注 GETSSH 扩展能力的开发者们强调：**v1.0 整个版本的一切重构，都是在为 v2.0 的 Plugin SDK 打地基。**

我们坚持“核心业务不妥协”的原则。SSH 终端与原生 SFTP 这“双擎”必须在极致安全、极致稳定的前提下运行。因此，我们必须先在 v1.2.0 中筑起高耸的防线（修复漏洞、剥离状态、建立沙盒），才能在后续版本中放心地将 API 钥匙交接给社区。

**在定档的 GETSSH v2.0 中，我们将正式推出：**
1. 完整定义的 `GETSSH_PLUGIN_API` 调用标准。
2. 允许第三方应用接管指定 UI 挂载点（如可视化面板、侧边栏动作）的渲染权限。
3. 丰富且安全的生命周期钩子（Lifecycle Hooks）及终端数据流拦截器。

## ⚠️ macOS 用户首次运行必读说明

由于 GETSSH 坚持独立开源的路线，尚未接入 Apple 官方的昂贵开发者证书签名体系，因此您的 macOS **Gatekeeper（网关拦截器）** 极有可能会在首次启动时进行安全拦截。

**解锁运行的两种方案：**

*   **常规 UI 方案**：请前往系统的 **“系统设置” -> “隐私与安全性”**，向下滑动找到 GETSSH 被拦截的提示，点击 **“仍要打开”** 即可。
*   **开发者终端方案 (推荐)**：若系统直接弹窗提示“应用已损坏，应将其移至废纸篓”，请打开终端 (Terminal) 执行以下命令，以强制移除 macOS 的隔离属性：
    ```bash
    sudo xattr -rd com.apple.quarantine /Applications/GETSSH.app
    ```
    *注：执行此命令需要输入您的 Mac 开机密码进行授权。*
